# 对比：Noeta vs Claude Agent SDK、LangGraph 与 Temporal

Noeta 和 Claude Agent SDK 都提供了代理循环、工具、MCP、子代理和会话。区别不在于这些表层功能，而在于底层的架构支柱。Noeta 让状态通过事件 fold 而成；SDK 则不是。

## 正面对比

| 服务端关注点 | Claude Agent SDK | Noeta |
| --- | --- | --- |
| **谁拥有底层基础设施** | Anthropic 托管模型；你在进程内托管循环 | 你拥有整个技术栈——循环、持久化存储、唤醒机制 |
| **状态 / 会话模型** | Session JSONL——追加式对话记录，自动持久化 | 事件溯源日志 + 内容寻址存储；状态 = `fold(events)` |
| **恢复 / 恢复执行** | 按会话 id 恢复或 fork；重放对话以继续 | fold 即恢复——重新 fold 日志即可恢复状态，无需单独的加载逻辑 |
| **挂起 / 唤醒** | 按会话 id 恢复 / fork；没有一等的持久化唤醒 | 一等持久化唤醒：被挂起的任务通过投影匹配并重新入队，恰好一次语义 |
| **上下文压缩** | 自动摘要，不可逆；可通过 `PreCompact` hook 拦截 | 压缩是一条被记录的事件——可审计、可复现，原始历史从不被清除 |
| **Provider** | 配置多个后端（Anthropic / Bedrock / Vertex / Azure），但以 Anthropic 为中心 | 厂商中立的内部协议；每个厂商在适配器之后；内核被禁止依赖任何厂商 |
| **工具** | 内置工具 + `@tool` + 进程内 SDK MCP server | 内置工具 + `@tool`（带 `version` / `risk_level`）+ MCP（stdio / HTTP） |
| **权限** | `permission_mode` + `canUseTool` + hook 链 | `permission_mode` + Guard（行动前鉴权） |
| **扩展** | Hooks，命令式拦截（`PreToolUse`、`PostToolUse`、…） | 五个扩展 seam（工具、策略、Guard、Observer、内容通道）加上单写者约束（Observer 只读） |
| **子代理** | 代理定义；输出返回给父代理；嵌套 ≤ 5 层 | 子任务是独立的事件溯源任务；扇出并发；结果通过 `SubtaskCompleted` 唤醒流回父任务 |
| **并发 / 分布式** | 单个 `query` / `Client` 进程内运行 | 基于租约 + 持久化日志的分布式队列底层（当前以单机形式交付） |
| **形态** | 一个直接调用 Claude API 的 TypeScript / Python 库 | 三个包——`noeta-runtime`（引擎）、`noeta-sdk`（客户端门面）、`noeta-agent`（多用户服务器平台） |

## 各自的适用场景

**选择 Claude Agent SDK 当**你想要开箱即用的代理循环，紧跟 Anthropic 官方能力，且运维负担最小。它是一个托管良好的客户端库——安装、指向你的 API key，即可使用。

**选择 Noeta 当**你需要掌控执行底层：每一步的持久化重放和审计、跨厂商的 provider 可移植性、将代理作为具备崩溃恢复能力的长运行服务托管，或从比任何单个进程更长寿的日志中 fold 状态。代价是你需要运行这套基础设施。

这两者与其说是竞争对手，不如说是对不同问题的回答。SDK 问的是"我如何给我的代码一个代理循环？"Noeta 问的是"我如何让代理的运行过程成为一份我可以重放、审计、带到别处的账本？"

> **坦诚的注意事项。** Noeta 是早期的 1.0 之前预览版。已交付的部署是单主机 / 单 worker 的；多主机协调和多 worker 防护尚未交付。生态系统更小——内置工具更少、没有插件市场、社区更年轻。如果"直接对接 Anthropic API 就能用"是首要需求，那么 Claude Agent SDK 是今天摩擦更低的选择。

## 三个差异详解

**事实来源的形态。** Session JSONL 也是一个追加日志，但它记录的是*对话*。Noeta 记录的是*事件*，状态是从这些事件中 fold 出来的投影。一个像是对话的录音；另一个像是状态机的账本。账本让恢复、压缩和审计都落在同一种机制上——恢复是一次重新 fold，压缩是一条被记录的事件，审计是另一次 fold。录音模型则需要为每一种操作单独准备一套逻辑。

**压缩的可逆性。** SDK 的自动压缩是摘要式的：原始内容被摘要替换，且过程不可逆（要归档得靠你自己在 `PreCompact` hook 里抓一份副本）。Noeta 的压缩只在日志中记录一条摘要边界；原始消息仍然存在，摘要在上下文组装时叠加。因此同一个任务恢复后会以相同的方式压缩——而且你可以挖出实际被削掉的内容。

**Provider 边界。** SDK 支持多个后端，但形态以 Anthropic 为中心——消息格式、工具调用约定、推理模型。Noeta 将内部协议设为厂商中立的规范，然后通过 import-linter 规则强制内核不依赖任何厂商 SDK。代价是每个厂商多一层适配器；回报是不绑定厂商的记录，以及你无需安装任何厂商 SDK 就能 fold 和审计任务。

## Noeta vs LangGraph

LangGraph 是最接近的开源邻居：它也持久化代理状态、支持人在回路中断、可以回退到更早的点（"时间旅行"）。区别在于持久化什么以及围绕它交付了什么机制。

| 关注点 | LangGraph | Noeta |
| --- | --- | --- |
| **持久化单元** | 每个超级步一个检查点：图的通道状态快照 | 只追加的事件账本；状态是 `fold(events)`，从不作为主副本存储 |
| **历史意味着什么** | 一组你可以回退或 fork 的状态快照 | 因果记录——每个事件携带 `actor` / `causation_id` / `trace_id`，因此历史回答的是*为什么*，而不仅仅是*什么* |
| **控制流** | 你定义节点和边的图；模型在其中路由 | 没有图。Policy 决定每一步；任务结构从决策中涌现 |
| **恢复 / 唤醒** | 调用方用恢复命令重新调用线程；队列和定时任务在托管 Platform 中 | Dispatcher + 租约 + worker 及持久化、恰好一次的唤醒在开源核心中交付 |
| **压缩** | 留给应用（或附加的记忆库） | 一条被记录的、可逆的事件——摘要在组合时叠加，原始内容保留在日志中 |
| **生态** | 成熟、集成目录庞大、社区大 | 年轻、工具集较小 |
| **Token 流式传输** | 通过图的事件 API 流式传输 | 作为临时 SSE delta 流式传输；事件账本始终是唯一的持久化记录 |

**选择 LangGraph 当**你想将代理表达为图，依赖 LangChain 集成目录，且今天就需要生态成熟度。

**选择 Noeta 当**问题在于可审计性和底层所有权：快照告诉你状态*曾经是什么*；账本告诉你*发生了什么*——哪个工具在谁的授权下运行、什么被压缩掉了、什么唤醒了一个沉睡的任务。而且调度机制（租约、持久化唤醒、崩溃回收）是开源核心的一部分，不是托管产品。

## Noeta vs Temporal（简述）

Temporal 是一个工作流引擎：你在代码中定义活动的 DAG，Temporal 持久化地调度和重试它们。它非常适合带有持久化定时器的人工编排业务流程。

Noeta 是一个代理运行时：LLM 动态驱动控制流，而非预定义的图。任务的结构从模型的决策中涌现，而非从工作流定义中产生。它们解决不同的问题——Temporal 用于当你提前知道工作形态时，Noeta 用于当模型在过程中发现它时。
