# Memory v2:政策 prompt、frontmatter 召回、memory_search、后台 consolidation

## Goal

在不改变 Memory v1 底座(file-per-memory 存储、`memory_write`/`memory_read` 工具、常驻索引、user-message seam 自动召回)的前提下,补齐三块能力:**记忆政策 prompt**(让模型知道什么值得记)、**召回升级**(frontmatter description 参与匹配 + grep 式 `memory_search`)、**后台 consolidation**(会话结束后异步整理记忆目录:合并重复、归档被推翻的、补写漏记的)。

## Non-goals

- **不上向量/语义检索。** v1 的 rule-of-two 决策维持(见 `docs/adr/unified-context-supply.md` Alternatives #7);行业实证(file + grep 路线)也支持这一点。未来若引入,走 D1 式 adapter 替换 `match_memories`,与本 spec 无关。
- **不做实时抽取管线。** 不在消息路径上用 LLM 抽 facts(Mem0 式);记忆形成只有两条通道:会话内模型自主调 `memory_write`,会话外 consolidation 补写。
- **不改存储位置与沙箱边界。** 全局目录(默认 `~/.noeta/memories`)、沙箱模式下留宿主机,均不动。
- **不做多索引分片、不做记忆跨 agent 共享。**
- **不物理删除记忆。** consolidation 与模型都只能 archive(可追溯),物理删除留给人。

## Context

- Memory v1 的设计与现状:`docs/adr/unified-context-supply.md`("Memory v1, four parts");代码在 `noeta.tools.memory`(store + 两个工具)、`noeta.context.memory`(索引 resident + `match_memories`)、`noeta.execution.memory`(recall seam)。
- v1 已知的三个 gap:召回只按记忆 **name** 的 token 匹配(summary 完全不参与);没有任何机制引导模型**何时**写记忆;没有整理机制,记忆只增不减、过时不失效。
- 2026 年中行业收敛(Anthropic memory tool / Claude Code auto-memory / OpenAI Dreaming / Letta MemFS):**文件存储 + 模型自管理读写 + 小索引常驻 + 后台异步整理**。本 spec 是把 noeta 从"有底座"补到这条主流形态的完整版,方向上无需推翻任何 v1 决策。
- 调研结论(benchmark 警示、各方案对比)不入库;若后续要选型评测,用 LongMemEval 并自跑 full-context baseline,不采信 LoCoMo 数字。

## Decisions

已与维护者确认的两项:

1. **Spec 范围 = 三个 phase 全含**,phase 1/2 可立即 implement,phase 3(consolidation)按本 spec 的决策实现。
2. **Consolidation 触发 = 会话停止 seam + 防抖**:交互会话没有 terminal 事件(正常停在 next-goal `suspended`,`close` 是 advisory 标记),所以触发挂在两个现有 seam 上——显式 close 级联与 turn 边界(drive 完成回到 suspended)——共用同一防抖检查:距上次整理超过阈值(默认 24h,marker 记录)才起任务。防抖使 turn 边界触发退化为"超过阈值后的第一个 turn 边界",等效周期行为而零新调度机制;不选定时器(空闲部署也会醒来)、不选纯手动(自动整理就不会发生)。marker 在入队时写,慢任务运行期间不会重复触发。

以下为按仓库惯例定的工程决策(实现时不再另行确认):

3. **Frontmatter 是可选的,params 生成,兼容旧文件。** `memory_write` 增加可选参数 `description`(一行摘要)与 `type`(枚举 `user | project | procedural | reference`),由工具组装 YAML frontmatter(`---` 围栏内 `key: value`),模型不手写围栏;`text` 自带 frontmatter 且未传 params 时原样接受。解析用限定在 `noeta.tools.memory` 内的极小解析器(只认围栏内 `key: value` 行),**不引入 yaml 依赖**。无 frontmatter 的旧文件行为不变(首行即 summary)。
4. **索引条目扩展为 3 元组,description 优先。** `MemoryStore.entries()` 产出 `(name, summary, type)`(type 缺省为空串);summary 取 frontmatter `description`,缺省回退首行(跳过 fence)。索引渲染在有 `type` 时输出 `- name (type): summary`,无则字节级维持 v1 的 `- name: summary`。`MemoryEntries` 别名同步为 3 元组——已确认它未经 `noeta.sdk` 公开导出,涟漪限于内部迭代点与测试;`match_memories` 签名不变,type 字段永不参与匹配。
5. **召回匹配升级为两档、保持确定性。** name-token 匹配(≥1 个 token 交集,现状)优先;其后 summary-token 匹配补位,**要求 ≥2 个不同 token 交集**(单 token 太噪);两档内部各按索引序,总量仍受 `DEFAULT_RECALL_MAX_HITS` 上限。纯函数,零 IO。
6. **搜索 = 新工具 `memory_search`,不复用 `memory_read`。** 单一职责 schema 对模型更稳;read 的 name-only 契约不动。语义:大小写不敏感的**子串匹配**(v2 不做 regex),范围 = name + 全文,输出 grep 风格(每条记忆最多 3 行命中摘录,最多 10 条记忆,确定性排序),不搜 archive。`MemoryStore` 增加 `search()`;工具进 `build_memory_tools` 的 pack,`risk_level="low"`。
7. **失效 = `memory_archive` 工具 + `archive/` 子目录。** `MemoryStore.archive(name)` 把文件移入 `root/archive/`;`entries()` 的非递归 `*.md` glob 天然排除子目录,索引与召回自动看不见。新工具 `memory_archive`(`risk_level="medium"`)进 pack——交互 agent(用户说"那条过时了")与 consolidation agent 用同一把刀。同名再 archive 采用带序号后缀防覆盖。
8. **政策 prompt 放 preset prompt 层,不进索引 resident。** 索引 renderer 在空 store 时零足迹(这是要保持的字节稳定性),而政策恰恰要在**第一条记忆写下之前**就在场,所以不能搭索引的车。做法:新增 prompt 片段(`presets/prompts/` 下独立文件),`main` 的 prompt 引入;片段同时作为公开常量导出,供自定义 spec(`memory=True`)的作者拼接。政策内容三段:什么值得记(用户纠正、跨会话项目事实、procedural 经验、外部资源指针)/什么不记(代码或 git 已记录的、只对本会话有意义的)/写前先查重(更新旧记忆而非另立新条,过时的 archive)。
9. **Consolidation 是普通 agent,工具面 = memory pack,无 fs/bash。** 一个内部 preset(不进用户可选列表),goal 里带宿主导出的"上次整理以来的会话摘录"(从 ledger 读 `MessagesAppended` 的文本块,近 N 个 session、每个截尾 M token,上限防爆);职责仅三:合并重复(写合并文件 + archive 原件)、archive 被推翻的、补写明显漏记的;单次运行操作数设上限。写入天然被 slug 约束限制在记忆目录内。
10. **防抖 marker = 记忆目录下 `.consolidation-state.json`**(非 `.md`,索引永远看不见),记录上次运行时间戳;触发点在 noeta-agent backend 的会话停止 seam,阈值可配,默认 24h。
11. **开关分层,与 instructions 同构:consolidation 不进 Capabilities。** 它是宿主行为而非 agent 身份——noeta-agent 产品层配置项(默认:memory on 则 on;env 可关),SDK 侧暴露显式入口函数供自行编排的宿主调用。`Capabilities.memory` 仍是记忆总开关。
12. **接受两项并发/新鲜度弱一致,写入文档。** ① consolidation 与活跃会话并发写:file-per-memory 使冲突面 = 单文件,last-writer-wins,consolidation 重写前重读原件;② 运行中会话的索引 resident 是 wiring 时快照,consolidation 后不刷新(`evolving` policy 本就允许 hash 漂移),召回路径读盘是即时的,新会话拿到新索引。
13. **长期文档:** consolidation 的触发/执行/失效语义是长期取舍,新增 `docs/adr/memory-consolidation.md`;`CONTEXT.md` 的 Memory 词条更新为 v2 四通道(write 工具 / 政策 prompt / 召回+搜索 / consolidation)。术语定为 **consolidation**(口语 dreaming 只在 ADR 里提一次作对照)。

## Implementation plan

### Phase 1 — 政策 prompt + frontmatter + 召回升级(纯增量,先发)

1. `noeta.tools.memory`:极小 frontmatter 解析(围栏内 `key: value`);`_first_line_summary` → `_entry_summary`(description 优先);`MemoryStore.entries()` 顺带产出 type(内部元组扩展或伴生访问器,以不动 `MemoryEntries` 公共形状为准——渲染层需要 type,匹配层不需要);`MemoryWriteTool` schema 增加可选 `description` / `type`,invoke 组装 frontmatter。
2. `noeta.context.memory`:`render_memory_index_text` 支持 `(type)` 标注;`match_memories` 两档匹配(name ≥1 / summary ≥2),cap 与顺序语义写进 docstring 与契约测试。
3. `presets/prompts/`:新增记忆政策片段;`main` prompt 引入;导出常量;`docs/reference/` 相应更新。

### Phase 2 — memory_search + memory_archive

4. `MemoryStore.search(query)`(大小写不敏感子串、确定性排序、行摘录)与 `MemoryStore.archive(name)`(移入 `archive/`、防覆盖后缀)。
5. 新工具 `MemorySearchTool` / `MemoryArchiveTool`,进 `build_memory_tools`;`docs/reference/tools.md` 及 zh 镜像更新。

### Phase 3 — consolidation

6. Ledger 摘录导出:宿主侧函数,自 marker 时间戳以来的 session 列表 → 每 session 的消息文本摘录(定界、截尾)。
7. 内部 consolidation preset(memory pack only)+ 运行入口(SDK 显式函数)。
8. noeta-agent backend:会话停止 seam 挂触发(读 marker、判防抖、经 background 子任务机制起 consolidation、成功后写 marker);产品层配置项 + env 开关。
9. `docs/adr/memory-consolidation.md` + `CONTEXT.md` Memory 词条更新 + releasing 相关 changelog。

## Acceptance criteria

1. **兼容性:** 无 frontmatter 的既有记忆文件,索引渲染、召回、read 的字节行为与 v1 完全一致;空 store 依旧零足迹(现有契约测试全绿)。
2. **frontmatter:** 带 `description`/`type` 写入的文件,索引行显示 description(与 type 标注);`memory_read` 返回全文含 frontmatter。
3. **召回:** name 命中优先于 description 命中;description 单 token 交集不触发;同输入同输出(确定性),cap 生效。
4. **搜索/归档:** `memory_search` 对 name 与正文子串命中、输出行摘录、不含 archive 内容;`memory_archive` 后该记忆从索引与召回消失,文件存在于 `archive/`。
5. **consolidation:** 会话停止且超过防抖阈值 → 后台任务运行且仅使用 memory 工具;重复记忆被合并(原件进 archive)、被推翻的进 archive、marker 更新;阈值内不触发;开关关闭时零行为。
6. **红线:** compose 路径无新增 IO;所有新内容进 context 仍走 ledger 先行(consolidation 不向任何活跃会话注入)。
7. `make check` 全绿;上述 1–5 各有契约测试覆盖。

## Risks

- **description 匹配引入召回噪声:** ≥2 token 门槛是猜的初值;若噪声仍高,先收紧为 ≥3 或要求含一个非常用词,不升级为语义检索。
- **consolidation agent 误归档:** 职责 prompt 需保守("不确定就不动");操作数上限 + archive 可逆是兜底;初期可在 noeta-agent 加运行日志便于人工抽查。
- **ledger 摘录体量:** 长会话多、摘录上限调不好会让 consolidation 变贵;N/M 初值保守(如 10 session × 4k token),配置可调。
- **会话停止 seam 的语义:** served 产品里"会话结束"可能表现为 idle 而非显式 close;T7 实现前先在 backend lifecycle/engine_room 确认停止事件的确切挂点,若只有 idle 语义则以 idle-timeout 为触发。
- **`MemoryEntries` 形状扩张的涟漪:** 渲染层要 type 而匹配层不要;若扩元组会波及既有签名,优先伴生访问器,避免无谓破坏。
