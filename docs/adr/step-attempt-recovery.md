# A mid-step crash is recovered per attempt: seal it with a snapshot-shaped marker, re-drive automatically when the approval surface proves it safe, otherwise park the task for a human

## Context

A worker crash mid-step (`kill -KILL`, power loss) leaves a **partial-step orphan**: the folded status is still `running`, but the live wake window ends in a partial decide→act attempt with no reachable suspend or terminal. One Step is a loop of such Attempts (see CONTEXT.md), and every attempt's first durable emit is `ContextPlanComposed` — so the stream already carries an implicit attempt journal (attempt identity = that event's seq; attempt intent = the plan plus the decision events after it). What was missing was a durable attempt-**abandon** record and a recovery posture.

The previous posture failed three ways. On the woken path the orphan was detected and raised as a typed `PartialStepOrphan`, but the worker's exception policy routed it into `fail(retryable=True)`: the task was re-leased, hit the same detection again, and after the retry budget went **silently terminal**. The drained path (an opening-turn crash, before any `TaskWoken` exists) had no detection at all and **silently re-drove on the dirty window**. And the prelude gap: a `202`-acked `send_goal` / `answer` wrote its prelude events later, on the drive thread, so a crash between the ack and that write **lost the user's acked input**.

Red lines: no dispatcher schema change and no change to wake-delivery semantics (the durable exactly-once machinery of subtask-fanout-and-durable-wake.md is untouched), and every existing recording must fold and resume byte-identically (event-sourced-truth.md).

## Decision

- **Recovery is classified-automatic (D1).** An interrupted attempt whose recorded activity is provably side-effect-safe is sealed and re-driven with no human involved; an attempt with unprovable side effects is sealed and the task is durably **parked** for a human decision. Never silently terminal, never silently duplicated. The seal records why: `"auto_redrive"`, `"unsafe_tool_activity"`, `"interrupted_approval"`, or `"abandon_cap"`.
- **One classification rule (D2): whatever could run without a human approval gate may be re-driven without a human.** Pure record events (plans, LLM request records, assistant messages, state patches, compaction) are always safe. Every recorded tool call — started *or* finished, because a re-drive re-decides and may re-execute finished calls too — is fed with its recorded arguments through the live guard chain (`guard_allows_tool_call` in `noeta.core.engine`: permission mode, risk ceiling, `can_use_tool`, skill grants — the same surface that gates live execution). Any non-ALLOW verdict means a human: a require-approval obviously, but a deny too, because the call's original side effects already happened once and only an operator can judge them. `permission_mode="bypassPermissions"` compiles to a gate-free chain, so under it every attempt classifies safe; whatever a mode auto-approves (e.g. edit-class under `acceptEdits`) classifies safe. A spawned subtask in the window, an unknown tool name, or a guard resolution failure all classify as needing a human (fail closed). Classification is evaluated against the **pre-attempt baseline** (the same bounded fold the seal serialises), not the dirty full stream: the D2 question is whether the re-drive — which runs on the sealed state — could proceed unattended, and counting the interrupted window's own activity into the Budget / Repetition counters could park an attempt the re-drive itself would allow.
- **The seal is a marker event + snapshot re-base (D3), not a journal table.** `StepAttemptAbandoned{abandoned_from_seq, state_ref, reason}` is the `TaskRewound` pattern scoped to one attempt: recovery folds the state to just before the interrupted attempt's anchor event (the same bounded point-in-time fold the conversation rewind uses), serialises it into the ContentStore, and appends the marker. fold treats it exactly like `TaskRewound` — an in-stream rebuild baseline, a member of the `find_latest_snapshot` set — so the partial attempt becomes folded-over dead history, still on the stream and still auditable. **Zero dispatcher change of any kind**; old recordings never contain the type, so there is zero byte drift; the payload is ~150 bytes, far under the 4 KB cap.
- **Only the interrupted attempt dies (D4).** The baseline is the folded state just before the interrupted attempt's anchor — its `ContextPlanComposed` (the *last* one of the live wake window), or for a plan-less approval window its first activity event. Completed attempts before it — plan, decision, fully paired tool starts/finishes — remain live history: their side effects are not re-executed and their results are not re-computed. Prelude events precede the first `ContextPlanComposed`, so the user's message survives automatically.
- **The case sentinel is `ContextPlanComposed`, not "any event after the wake" (D5).** The worker's woken reconciliation on a `running` status keys on whether an attempt started: none ⇒ **case 2′**, run the bare step — now correct by construction for every caller (timer/subtask re-deliveries, seeded command wakes whose prelude is already durable, and a crash that landed between a seal and its re-drive); present ⇒ **case 5′**, the recovery machine: classify → seal → re-drive under the same lease, or park. The drained branch gets the identical treatment (a `running` drain scans before stepping), closing the silent dirty re-drive. The `PartialStepOrphan` exception class is deleted — a partial attempt is no longer a fail path anywhere. One orphan is plan-less by construction: an **interrupted approval execution** — the drive-side approval prelude executes the approved tool before the step's first plan, so a crash there leaves activity (`ToolCallApprovalResolved`, a `ToolCallStarted`) with no `ContextPlanComposed`. The scanner anchors that attempt on the first activity event instead, and recovery **always parks it, never re-drives** — a human is in that loop by definition, and duplicating an approved side effect without asking is exactly what recovery exists to prevent.
- **Append-type preludes become durable at seed time (D6).** Preludes that only append events (`send_goal` message, `answer`, a chained `ModelBound`) are marked `durable_at_seed` and applied synchronously on the request thread: `note_woken` plus the prelude events land before the ack, so `202` now means "your input is durable", and the drive enters the woken machine prelude-less at case 2′. Event order and bytes are unchanged — only the wall-clock moment of the write moves. The seed resolves the Engine **before** applying the prelude and pins it on the seeded turn, so a seed-written `ModelBound` keeps its drive-the-next-turn `/model` semantics; the accepted divergence is that a crash re-drive of a model-switch turn re-resolves from the log and runs on the **new** binding. Both lease-acquisition paths (first try and the dispatcher-restore retry) share one seed tail, so the durability promise cannot silently depend on which path produced the lease; and a prelude failure *after* `note_woken` (e.g. a payload-cap error) compensates by re-suspending on the command's own handle and releasing the lease — the conversation stays resumable and the client's retry of the same command works, instead of stranding a half-seeded `running` window the worker would later re-drive without the input. The approval prelude stays drive-side (it *executes* the approved tool and must not block the request thread); a crash before the resolution lands is benign — the task re-suspends on the same approval and the operator approves again — and a crash *during* the approved execution is the plan-less window above, which parks.
- **Park = a stopped conversation (D7).** Under the recovery lease: seal → append one `origin="system"` notice naming each interrupted call and whether it completed → suspend on `NEXT_GOAL_WAKE_HANDLE` (hoisted to `noeta.protocols.wake`) → release suspended, consuming the wake. The parked task is exactly a stopped conversation: the web UI shows it suspended with the notice, typing a message resumes it from the clean baseline, and `close` / `cancel` work as-is. **Zero new API verbs, zero new UI forms — no operator verb exists for a parked task at all.** The one variation: an interrupted approval execution re-suspends on the window's **own** wake handle (`approval-<call_id>`) instead of the next-goal handle — pending approvals are folded state, so the seal restores the pending approval and the ordinary `approve` / `deny` verbs simply work again; the operator re-approves after reading the notice. The handle comes from the wake the lease consumed; a wake-less re-entry recovers it from the stream (the suspension the window consumed).
- **Repeated crashes recurse, capped at `ABANDON_CAP = 3` (D8).** The seal is deliberately *not* a wake-window boundary for the woken matcher (unlike `TaskRewound`), so a crash during a re-drive re-enters recovery: before the seal as a fresh case 5′; after an `auto_redrive` seal as a bare case-2′ re-drive (the seal already re-based the state); after a **park-reason** seal the re-entry completes the park instead (`scan_pending_park`) — the reason recorded in the seal is the durable "do not re-drive" decision, so the remaining writes (the notice, skipped when it already landed; the suspend; the release) are finished under the new lease rather than a bare step running over them. Blockers for a re-appended notice are re-derived from the sealed dead tail. Consecutive seals within one window are counted, and past the cap recovery seals with reason `"abandon_cap"` and parks regardless of classification — the crash-loop backstop, mirroring the dispatcher's `reclaim_max`. When a plan-less approval window and the cap coincide, `"interrupted_approval"` wins the reason: the notice must tell the operator a human-approved call was interrupted, whatever the seal count.
- **Auto re-drive is silent; only a park speaks (D9).** A safe re-drive just re-runs — injecting a notice would perturb every recovered transcript for no benefit. The park notice is the only transcript change, and only on new recordings.

## Rationale

- **Classified-automatic is the only posture that closes both failure modes at once.** "Manual with a typed error" had already decayed into silent termination in practice, and the overwhelmingly common orphan (a reads-only or plan-only window) needs no human judgment at all. The actual hazard is *duplicate side effects*, and that is exactly what the classification isolates: automate everything below the approval line, stop dead above it.
- **The classification borrows an already-trusted boundary instead of inventing a second safety vocabulary.** "Would this call run unattended live?" is a question the guard chain already answers for every tool call; reusing it means recovery can never be more permissive than live execution. The corollary is the same trust surface, including its edge: a side-effectful custom tool left at the `@tool` default `risk_level="low"` classifies as safe — but such a tool already runs unattended live today, so this is not a new exposure. Authors of side-effectful custom tools must declare `medium` / `high`.
- **The classification is self-consistent with how a window gets orphaned in the first place.** A plan-anchored attempt can only contain tool calls the guard allowed unattended — an approval-needing call would have suspended the window rather than orphaned it — so plan-anchored attempts are auto-redriveable almost by construction. The `"unsafe_tool_activity"` parks are the degenerate cases (guard configuration drift between the original run and recovery, a spawn in the window, a guard resolution failure), and the windows with a human's risk in them are exactly the approval executions — which always park.
- **A marker + snapshot re-base reuses the entire rewind machinery and touches no hardened surface.** The attempt journal already exists implicitly (`ContextPlanComposed`); only the abandon record was missing, and one snapshot-shaped event supplies it while inheriting the fold reset, the baseline-set lookup, and the bounded point-in-time fold wholesale. A dispatcher-side journal would add mutable state that replay must reproduce and would cut into the hardened wake-recovery incision — for state that is derivable from the stream.
- **Attempt granularity is what makes automatic recovery honest.** Sealing the whole window would re-execute completed attempts' side effects, re-compute their results, and discard the turn's prelude (the user's message). Sealing only the interrupted attempt keeps everything provably-done alive and re-decides only what was in flight.
- **The seal is written under the recovery lease, so it is fenced.** Unlike the control-plane `TaskRewound`, `StepAttemptAbandoned` is a lease-checked append: a zombie step thread surviving from the crashed process cannot race the seal, and nothing in the machine assumes a worker count.
- **Seed-time preludes turn an ordering promise into a durability promise.** The bytes and their order were already right; only the ack raced the write. Moving the append to the request thread makes the `202` mean what callers assume it means, and it is also what makes case 2′'s bare re-drive *provably* correct rather than correct-for-known-callers.
- **Park-as-stopped-conversation costs nothing to learn or build.** The suspended-on-next-goal shape already exists for a human stop; a parked task inherits resume, close, and cancel for free, and the notice does double duty — the human reads it in the UI while the task rests, the model reads it when the conversation resumes.

## Alternatives considered

1. **A dispatcher-side attempt journal table.** Rejected: it puts recovery state into the dispatcher — one more mutable state that replay must reproduce — and touches the hardened `matched_wake_event` wake-recovery surface, when the attempt journal is already derivable from the stream (`ContextPlanComposed` is the start record; only the abandon marker was missing).
2. **Whole-window abandon (re-base to the turn boundary).** Rejected: it re-executes completed attempts' side effects, re-computes their results, and throws away the turn's prelude events — the user's message would die with the window. Attempt granularity keeps all of that live for the price of one bounded fold.
3. **Wake-event-carried command intent (put the command payload into the wake event so a re-delivered wake replays it).** Rejected: it grows dispatcher-held wake state into a second carrier of conversation content, needs its own dedup story on re-delivery, and violates the single-truth rule — user input lands in the EventLog, nowhere else. The seed-time write achieves the same durability with zero shape change.
4. **Staying manual, with better tooling only (keep the typed error, add inspect/re-drive verbs).** Rejected: it leaves the acked-input loss and the silent-terminal decay unaddressed, imposes operator work on the common all-safe window, and adds a verb surface for a decision the recorded data already answers.

## Consequences

- Protocol and fold: `StepAttemptAbandonedPayload` lands in `noeta.protocols.events` (its restore mapping in `noeta.storage._payload_restore`, like every payload); the fold reset (shared with `TaskRewound`) lands in `noeta.core.fold`; the `find_latest_snapshot` baseline set is defined once as `SNAPSHOT_BASELINE_EVENT_TYPES` in `noeta.protocols.event_log` and consumed by every adapter's lookup (`noeta.storage.sqlite`, `noeta.storage.postgres`, `noeta.storage.memory`) and by `BoundedEventLog` — the migrations' index predicates stay frozen literals, pinned in sync with the constant by the query-plan test. Two hoists: `NEXT_GOAL_WAKE_HANDLE` moves to `noeta.protocols.wake` (re-exported from `noeta.execution.multi_turn`), and `BoundedEventLog` moves to `noeta.core.fold` (aliased in `noeta.execution.driver`) so the seal's point-in-time baseline fold and the rewind share one implementation.
- Runtime: the pure attempt scanner + classifier and `ABANDON_CAP` land in `noeta.runtime.attempt`; the seal / re-drive / park machine, the case-2′/5′ sentinel, the drained-branch routing, and the `attempt_abandoned` / `attempt_parked` reliability kinds land in `noeta.runtime.worker`; the lease-checked `abandon_step_attempt` and the classifier's `guard_allows_tool_call` predicate land in `noeta.core.engine`.
- Execution: seed-time prelude application (`durable_at_seed`) and the pre-prelude Engine pinned on the seeded turn land in `noeta.execution.driver`.
- Observability: the OTLP exporter segments the task trace at the seal exactly as it does at a rewind, and the web trace timeline renders the marker.
- Storage migration: sqlite migration 8 and postgres migration 2 re-create the partial fold-baseline index with the widened `IN ('TaskSnapshot', 'TaskRewound', 'StepAttemptAbandoned')` predicate — a partial index is only chosen when its `WHERE` matches the query exactly.
- Costs and limits: the seed lengthens the command's request path by a few bounded appends; recovery prevents silent *duplicate* execution but compensates nothing — inspecting what a crashed side-effectful call actually did remains the operator's job, and the park notice says exactly that; a model-switch turn re-driven after a crash runs on the new binding (accepted divergence noted above).
- For the multi-host fencing decision to come: recovery always runs under a lease and the seal is a lease-checked append, so a fencing epoch that fences leases automatically fences recovery too — nothing in this design assumes a single worker.
