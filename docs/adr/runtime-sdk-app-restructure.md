# runtime/sdk/app re-layering: noeta-runtime is the pure engine, noeta-sdk is a thin in-process client, noeta-agent owns HTTP/SSE, noeta.sdk is the sole public surface, and the front/back protocol is reset

## Context

This decision evolves from `docs/adr/library-sdk-architecture.md` (the earlier three-layer runtime/sdk/product decision) and **partly supersedes** it. The physical import topology is in `docs/adr/package-layout.md`; the provider-neutral kernel↛adapter contract is in `docs/adr/provider-neutral.md`; the identity / provenance machinery is in `docs/adr/agent-identity-and-provenance.md`; the event-envelope stream and fan-out neutrality are in `docs/adr/event-sourced-truth.md` and `docs/adr/transport-neutral-fanout.md`.

This document supersedes the part of `library-sdk-architecture.md` that layered by **mechanism vs material**. **Removed**: the split where sdk held "the concrete implementations of the protocols (policies+tools+providers+context)" while runtime held the protocol contracts (peers of the kernel and kernel services); the inversion that "lifted the execution machinery into sdk"; "putting the server into sdk"; and the user-facing requirement to "replace an entire `ContextComposer` wholesale" (the Options-level seam for replacing the composer wholesale). This decision adopts two options that document had previously rejected in its own alternatives section — its #1 (material — **together with** the identity layer — sinks into runtime, and sdk keeps only the client surface) and its #3/#4 (server goes into product; composer stays in runtime with its internals, not lifted out) — and it dissolves the old objection ("sdk becomes an empty shell") using a public-surface boundary (the `noeta.sdk` facade) rather than a material/mechanism boundary.

**Preserved** from `library-sdk-architecture.md` and its sibling decisions: provider-neutral (kernel↛adapter); `AgentSpec` structural-equality identity + pure versioned wiring factory + `AgentBound` provenance; the single-writer Engine; event-sourced fold/resume; stable-prefix cache-friendliness — **these five invariants do not move.** The `Options`→frozen `AgentSpec` compilation (additive identity, not a replacement layer) is likewise preserved as a decision. What the re-layering moves is dists, not import paths.

## Decision

### Three layers, redefined

Distribution is still three wheels, but the boundary is now **the outer wheel + the public surface**, not mechanism vs material. The model is **in-process, like LangChain** — no subprocess and no mandatory HTTP between the caller and the engine.

```text
noeta-runtime  the pure engine: carries/records/schedules a Task, folds the EventLog, and contains all agent-authoring material
noeta-sdk      the thin in-process client: the public facade, re-exports + authoring API, no engine, no HTTP
noeta-agent    the product: a new in-process Python backend + the apps/web frontend + HTTP/SSE
```

- **`noeta-runtime` absorbs the material.** `policies`, `tools` (everything except the authoring API `@tool` / `create_sdk_mcp_server`), `providers`, `context`, `execution` (`runner` / `driver` / `resolver` / `multi_turn` / `subtask_drain` / `builder`), the agent **identity** layer (`agent.spec` / `agent.registry`), and `presets` all live in the runtime wheel. Runtime's internal **8-layer import topology is preserved**: import paths don't change, only wheel membership does. The move in `library-sdk-architecture.md` that lifted the execution machinery upward is **inverted** — runner / driver / resolver sink into the engine rather than into sdk; the engine is what runs an agent, and the machinery belongs to it.
- **`noeta-sdk` is the thin client.** It holds the `client` public surface, the authoring API (`@tool` / `create_sdk_mcp_server`), the re-exports of the extension protocols, and the re-export of `presets`. It contains **no engine and no HTTP** — every call delegates in-process into runtime.
- **`noeta-agent` (under `apps/`) is the product.** A new Python backend that imports **only** `noeta.sdk`, plus the `apps/web` frontend, plus HTTP/SSE transport. It defines **no agents of its own**; the four-preset roster ships from runtime's `presets`, re-exported through sdk.

### Public surface and encapsulation: `noeta.sdk` is the only door

- **Users install only `noeta-sdk` and import only `noeta.sdk`.** `noeta-runtime` is a transitive dependency they never touch; its internal modules are not part of the sdk API surface.
- **The public export set (defined once).** `noeta.sdk` re-exports: `query` / `Client` / `Options` / `AgentDefinition` / `SystemPromptPreset` / `tool` / `create_sdk_mcp_server` / `as_messages`; the extension protocols `Tool` / `LLMProvider` / `Policy` / `Guard` / `Observer` / `ContentKindSpec` (plus `View` / `Decision`, labeled advanced extension contracts); and the `presets` roster. `noeta.client.*` (including the internal symbols `compile_options` / `messages` / `parts`) is the implementation behind the facade, not part of the public surface.
- **`import-linter` welds shut the in-repo half.** A new `forbidden` contract, `app-uses-only-sdk`, stops product modules (`noeta.agent.api/host/commands/observe/read_models/skills` + the new HTTP/SSE backend modules) from importing any runtime internal: `noeta.core`, `noeta.protocols`, `noeta.runtime`, `noeta.context`, `noeta.policies`, `noeta.tools` (the whole package — `@tool` / `create_sdk_mcp_server` ship in `noeta.sdk`, not `noeta.tools`), `noeta.providers`, `noeta.execution`, `noeta.agent.spec`, `noeta.agent.registry`. The app crosses the boundary only through `noeta.sdk`. **Deliberately exempt** (the app may import these directly): `noeta.storage` (the host wires up a concrete backend) and `noeta.read_models` (peripheral resource projections: file tree / preview). The storage exemption is for **wiring** only — the app drives state changes only through `noeta.sdk` / `Client` verbs and never calls `EventLog.append` / `system_emit` directly, so the Engine remains the sole writer.
- **sdk→runtime imports are legal** (in-process is fine); the linter only forbids **app→runtime internals**. The "external user code imports only `noeta.sdk`" half is a refinement of the D2 intent, and since `import-linter` can't reach across wheel boundaries, it is guaranteed by **wheel packaging**: users install only the `noeta-sdk` wheel, whose API surface is `noeta.sdk` + the authoring API; runtime modules ship in the transitive `noeta-runtime` wheel and are not part of that surface.

### Extension surface: open via `Options`, narrow the Composer, lock down the Engine

- **The open extension points are all `Options` fields**: `Tool`, `LLMProvider`, `Policy`, `Guard`, `Observer`, and content channels (registered via `ContentKindSpec`). `provider` is in this open band too (provider-neutral is the foundation).
- **The storage backend** (`EventLog` / `ContentStore` / `Dispatcher`) is open, but configured by the **host** rather than `Options` — through the `noeta.sdk.HostConfig` surface (`Client(..., host_config=HostConfig(...))`). The same surface carries the host's other non-identity runtime injections (`app_gateway` the preview gateway, `mcp_server_resolver` for live MCP alias resolution, `mcp_http_post`), keeping them all outside `AgentSpec` identity (`compile_options` never sees `HostConfig`). Storage defaults to in-memory; these injections are absent by default (no `open_app`, no live MCP), so a bare `HostConfig()` is byte-identical to today.
- **`ContextComposer` is narrowed on the user surface only.** The Options-level entry point for replacing the composer wholesale is removed; the only context extension is `Options.content_channels` (registering a `ContentKindSpec`). Internally the composer is **still injected at the engine seam via a Protocol**: `noeta.core.engine` imports only `noeta.protocols.composer.ContextComposer` the Protocol, `noeta.core` keeps its pure-protocol `PassthroughComposer` fallback, and `ThreeSegmentComposer` is wired by the builder in the runtime band — there is **no** default `noeta.core → noeta.context` import, so `core-uses-only-protocols` still holds. The user can't replace the composer because it is a hard constraint of stable-prefix prompt caching (see `docs/adr/context-compaction.md`, `docs/adr/unified-context-supply.md`).
- **The Engine main loop and `Dispatcher` / `Worker` / `Lease` are locked down** — there is no code-replacement seam. The only knobs are concurrency and lease duration in the host config (see `docs/adr/worker-lease-model.md`, `docs/adr/engine-policy-dataflow.md`).

### `Options` semantics: replace-style tools, no `add_dirs`

- **`allowed_tools` is replace-style.** Setting it means "**only** these"; not setting it = the full official default set of **11** tools (`read` `glob` `grep` `edit` `write` `apply_patch` `shell_run` `shell_poll` `shell_kill` `webfetch` `web_search`). `disallowed_tools` is **always subtractive**. `memory_read` / `memory_write` and the control tools (`todo_write` / `skill` / `ask_user_question` / `spawn_subagent`) are only attached under their corresponding capability. This corrects the additive `allowed_tools` in `library-sdk-architecture.md`.
- **No `add_dirs`.** The remaining `Options` fields (`permission_mode`, `can_use_tool`, `max_turns`, `cwd`, `system_prompt`, `agents`, `skills`, `provider`, plus the advanced `name` / `budget` / `capabilities`) are kept as-is; the full table is in the implementation spec. The compiled `AgentSpec`'s **structural identity** reflects the concrete tool list resolved after compilation (component lists normalized to sorted tuples); `model` is a binding (`ModelBound`) and `provider` is wiring — neither enters that identity. (The earlier deterministic `fingerprint` digest retired along with verify/replay; identity is now the structural equality of a frozen `AgentSpec`, see `docs/adr/agent-identity-and-provenance.md`.)

### HTTP/SSE belongs to the app

The HTTP/SSE server lives **only** in the app layer. `noeta-runtime` and `noeta-sdk` are pure in-process libraries with no transport, aligned with `claude-agent-sdk` / LangChain (a library doesn't bundle a server). The official app eats its own dog food: a thin backend (`noeta/agent/backend/` — `engine_room.py` drives the engine only through `noeta.sdk`, `app.py` owns the HTTP server + routes, `lifecycle.py` assembles the host config, `stream.py` / `task_protocol.py` carry the core SSE + command protocol, `resource_services.py` / `mcp_service.py` / `read_views.py` carry the peripheral data surface + index views) wraps `noeta.sdk`; a third party copies this pattern. No shared server package is extracted.

### Front/back protocol: one multiplexed envelope stream + command verbs + separate resource services

The front/back interface is reset. The **core task protocol** (small and hard) is separated from the **peripheral resource services**.

- **Core = one multiplexed SSE envelope stream per conversation.** It carries the raw `EventEnvelope`s of the root task plus all subtasks; the frontend demultiplexes by `taskId`; the envelope's `seq` is the SSE event's `id`, used for `Last-Event-ID` resume. Plus a small set of **command endpoints aligned with `Client` verbs**: start / send_goal / approve / deny / answer / cancel / close / reopen. Commands return `202` / ack (with a `taskId` when needed) and are idempotent; **all results surface through the envelope stream as the single source of truth**. Each subtask having its own stream is a hidden internal fact of the backend — the backend merges them.
- **Peripheral = independent endpoints**, never mixed into the task protocol: `GET /content/{hash}` (`?mode=raw`) fetches a content block by hash, `/files?task` + `/file?task&path` fetch the workspace tree / preview, `/preview/<token>/` serves the app preview gateway (single port, relative addresses, `sandbox="allow-scripts"`), `/mcp/*` serves MCP management (CRUD + per-session enable + prompts/resources). Large objects (images / big files / tool artifacts) travel through the stream as just a `ContentRef`; the body is fetched on demand from `/content/{hash}`. **Image input** (paste / choose image in the composer) is written through `noeta.sdk` — `Client.put_content(bytes, media_type) → ContentRef`, wrapped in the user turn's `ImageBlock` — and read back from the same shared `/content/{hash}`. The new backend **deliberately does not** reproduce the old host's per-task scoped image endpoint (`/tasks/{id}/images/{hash}` + the A2 "only structurally referenced refs" boundary of `collect_task_image_refs` in `docs/adr/web-image-attach.md`): noeta-agent is single-user local, the content hash is itself a 256-bit capability, and `/content/{hash}` already serves every kind of artifact — scoping images separately closes off no real surface and merely duplicates a read path the frontend already uses.
- **Index views** (`read_views.py`) are read-only projections the UI shell needs outside any single stream: `GET /capabilities` (composer enumeration — agents / models / permission+effort modes / mcp servers + per-model vision gate, projected only through `noeta.sdk`) and `GET /tasks` (the list of root conversation sessions, with `status` / `closed` / `title` folded from each root's stream; subtasks filtered out). These are thin server-side index projections (like the file tree), explicitly allowed for non-event views — per-conversation UI state is still folded in the frontend (D7).
- **Persistent storage** is opt-in via the `HostConfig` `(event_log, content_store, dispatcher)` triple. `serve_backend` builds a durable triple when `BackendConfig.storage_url` / `NOETA_AGENT_STORAGE` is set — a sqlite file path or a `postgresql://` DSN, dispatched by `noeta.agent.host.storage.open_durable_storage` (host-side material; the legacy `sqlite_path` / `NOETA_AGENT_SQLITE` spellings remain accepted) — so conversations + the `GET /tasks` list survive restarts; unset ⇒ the SDK's in-memory default (single run, enough for in-process preview / MCP). The app never writes events directly — the persistent store is populated only through `Client` verbs (the Engine remains the sole writer).
- **Hard constraints**: the wire carries `EventEnvelope` **as-is** (no invented `SessionEvent`); everything is addressed by **`taskId`** (no separate `sessionId` — "session" is just the runner's name); single-user, local, no auth / tenancy.

### The frontend folds the raw envelope stream

UI state is derived by **the frontend folding the raw envelope stream** (`apps/web/src/domain/reducer.js`, accumulating by `seq`, bucketed by `taskId`). The reducer derives only **display state**: the authoritative fold/resume is still the backend's `noeta.core.fold(EventLog, ContentStore)`, `EventLog` is still the single source of truth, and the reducer must obey the same append-only `seq` order and `EventEnvelope` semantics. The backend stays thin and **maintains no per-UI pre-projection**. `read_model` survives only for peripheral resources (file tree / preview), which is the allowed pre-projection exception.

### The key insight: what moves is the wheels and the public surface, not the module topology

This change touches the **outer wheel boundary + the public surface**. Runtime's internal **module-level import topology is preserved** — `import-linter` constrains "which module imports which module," not "which wheel ships it." `noeta.runtime ↛ noeta.providers` (the provider-neutral kernel↛adapter contract) still holds even though `providers` now ships in the `noeta-runtime` wheel, because PEP 420 keeps the import path `noeta.providers.<name>` stable and only the physical dist moves. Import paths (`noeta.policies.*` / `tools.*` / `providers.*` / `context.*` / `execution.*` / `agent.spec` / `registry`) stay in place to minimize churn; only wheel membership changes. The dist names `noeta-runtime` / `noeta-sdk` / `noeta-agent` are unchanged.

The result: a user does `pip install noeta-sdk`, `from noeta.sdk import query, Client, Options, tool`, and runs an agent fully in-process; the product is the same library plus a transport layer, and it can't reach any engine internal across the public boundary.

## Rationale

- **The mechanism/material cut drew the boundary in the wrong place.** It forced sdk to become a **fat material library** (all policies / tools / providers / context) rather than a thin client, and left runtime unable to run an agent itself (the execution machinery was stuck above it). Cutting by **outer wheel + public surface** lets runtime become a truly self-sufficient engine and sdk a thin door — which is what a library SDK is supposed to look like.
- **The execution inversion is fixed by sinking, not by lifting.** `library-sdk-architecture.md` **lifted** runner / driver / resolver into sdk so the library could run an agent, which made sdk fat. **Sinking them into runtime** achieves the same "the engine runs the agent" goal while keeping sdk thin.
- **This aligns with `claude-agent-sdk`'s thin-client identity.** The official Claude Agent SDK and LangChain are in-process libraries and do **not** ship a server; transport is owned by the consumer. Making noeta's public surface the thin `query` / `Client` / `Options` / `tool` layer, with the engine as a transitive dependency, makes noeta a genuine library rather than "a product you must import wholesale."
- **HTTP belongs to the app, not the library.** The server is a deployment choice. Pushing it into `noeta-agent` means the official product eats its own dog food through `noeta.sdk`, and any third party hosts its own agent by copying a thin backend — dragging in no coding-tool baggage.
- **A public-surface boundary beats a material boundary on encapsulation.** The old objection "sdk becomes an empty shell" is answered: the shell **is** the API. `import-linter`'s `app-uses-only-sdk` plus wheel packaging weld the encapsulation shut, so internal modules can be freely reorganized without leaking into the public surface.
- **Frontend folding keeps the backend honest.** The envelope stream is already the canonical fold input; having the frontend fold it (rather than maintaining a server-side conversation projection) means exactly one source of truth and no projection that can drift. `read_model` is kept only where a resource genuinely needs server-side shaping (file tree / preview).
- **The five invariants survive because they are path/value properties, not dist properties.** provider-neutral is two path-based forbidden contracts; identity is a property of frozen `AgentSpec` values; single-writer is "only `noeta.core.engine` appends to `EventLog`"; fold is a pure function of `(EventLog, ContentStore)`; stable-prefix emerges from canonical recording + pure composition. Moving code between wheels doesn't touch any of these.

## Alternatives considered

1. **Pure in-process facade (A) — drop the wheel boundary and ship one package.** Rejected: it drops the boundary discipline. Without the `noeta-sdk` / `noeta-runtime` wheel split + `app-uses-only-sdk`, nothing stops product or user code from reaching into engine internals, and the public surface erodes back to "import whatever you want."
2. **Force a real process boundary (B) — the engine always runs in a separate process behind a socket.** Rejected: a local single-user product would pay a daily two-process tax (serialization, lifecycle, IPC) for a boundary it doesn't need. In-process is the default.
3. **Pluggable transport / server-in-sdk (C, the remote form).** Rejected for now (deferred): putting the transport seam or server back into the library would re-fatten sdk and re-introduce the inversion. The remote / cross-process form is left for future extension; this round is in-process only.
4. **Borrow an off-the-shelf open-source chat frontend** (derived from the non-goal). Rejected: it is impedance-mismatched with noeta's task/event model. An open-source frontend assumes a message/session shape; noeta's UI is a fold over `EventEnvelope`s addressed by `taskId` with subtask multiplexing, which a generic chat UI can't express and would fight everywhere.
5. **Backend pre-projection / server-side `read_models` for the conversation view** (derived from D7). Rejected in favor of frontend folding. Maintaining a per-UI projection on the server would duplicate the fold the frontend already needs for resume, creating a second source of truth that can drift. `read_model` is limited to peripheral resources.
6. **Copy Claude's async + pure message stream** (inherited from `docs/adr/library-sdk-architecture.md`). Rejected: it is impedance-mismatched with a synchronous, event-sourced, single-writer engine; it would demote the canonical envelope stream to an internal detail.
7. **Keep a separate CLI launcher layer** (inherited from `docs/adr/library-sdk-architecture.md`). Rejected: the product form is an agent with a frontend; a thin `python -m noeta.agent` entry point plus command endpoints aligned with `Client` verbs is enough.

## Consequences

- **Wheel layout**: `packages/noeta-runtime` (the pure engine + the absorbed `policies` / `tools` / `providers` / `context` / `execution` / `agent.spec` / `agent.registry` / `presets`), `packages/noeta-sdk` (the thin client), `apps/noeta-agent` (the product + HTTP/SSE). The physical topology details are in `docs/adr/package-layout.md`.
- **The public facade** is `noeta.sdk`, re-exporting the export set defined in the "Public surface" section; `noeta.client.*` is the internal implementation behind it.
- **`noeta.client.options`** carries the `Options`→frozen `AgentSpec` compilation, replace-style `allowed_tools`, and no `add_dirs`; identity is unchanged (see `docs/adr/agent-identity-and-provenance.md`).
- **`noeta.execution.*`** (`runner` / `driver` / `resolver` / `multi_turn` / `subtask_drain` / `builder`) is in the runtime wheel and is the only canonical construction point; it resolves registered `(name, version)` refs and errors loudly when a component is missing / drifted.
- **`noeta.core.engine`** is the sole physical writer; it imports only the `ContextComposer` Protocol (the composer is wired by the builder, not in core); `Dispatcher` / `Worker` / `Lease` are locked-down seams (see `docs/adr/single-writer-invariant.md`, `docs/adr/worker-lease-model.md`).
- **The thin product backend** is `noeta.agent.backend` (`engine_room.py` / `app.py` / `lifecycle.py` / `stream.py` / `task_protocol.py` / `resource_services.py` / `mcp_service.py` / `read_views.py`), which imports only `noeta.sdk` and owns the multiplexed SSE envelope stream + command endpoints + peripheral resource services (content/files/preview/MCP) + read-only index views (capabilities/session-list); `apps/web/src/domain/reducer.js` (+ `multiplex.js`, bucketed by `taskId`) folds the raw stream into display state, including the `wakeKind` / `closed` details the send gate reads. Persistent storage is host-side material living in `noeta.agent.host.storage` (the `HostConfig` sqlite triple), a peer of `preview_gateway` / `mcp_registry`.
- **The current state of the `import-linter` contracts**: the full public seal `forbidden` `app-uses-only-sdk` is **not yet in effect**, and T8 exposed why it can't be flipped on mechanically. T8's deletion set contained two different kinds of thing: **(A) the old HTTP serving stack** — `noeta.server.*` + the superseded `noeta.agent.host.{http*,server*,runner_cli,request_models,oneshot,content_resolver,provider_headers,session_store,workspace_*,mcp_prompt_expander}` — and **(B) the engine-assembly layer** — `noeta.agent.host.session` (the 1342-line `AgentSessionRunner` / `CodeSessionConfig`) + its `noeta.agent.api` facade. **A was deleted in T8/R10; B was deleted in T8/R11**, and before deleting B its ~50 engine-behavior tests (approval / apply_patch / subagent / workflow / skills / MCP…) were first migrated onto the production `SdkHost` + `InteractionDriver` path via the `tests/_sdk_session` harness — `noeta.sdk.Client` / `Options` can't express the host knobs those tests need (`require_approval_tools` / stdio-MCP / `hooks` / `skill_tool_enforcement`), so the migration target is `SdkHost` (which exposes them as fields and reads per-agent flags from `spec.capabilities`). The migration exposed two runner-only ADR features missing on the SDK path — memory recall (`driver.seed_start` does plain append only, with no per-goal recall / index event) and per-helper workflow structured output (`GenericEngineResolver._build_subtask_engine` drops the child `output_schema`) — whose tests are `pytest.mark.skip` (with a reason) awaiting the port into the SDK seam. Even after B is deleted, the full seal is still blocked: the host **material** kept for the new backend — `preview_gateway` (→ `noeta.tools.app`), `mcp_registry` (→ `noeta.tools.mcp`), `storage` (→ `noeta.storage.sqlite` / `noeta.protocols`) — legitimately imports runtime internals directly, so an `app-uses-only-sdk` covering all of `noeta.agent` would cause collateral damage until those modules are either routed through `noeta.sdk` or relocated back under the app. The contract in effect meanwhile is `backend-only-sdk` — same intent, but with `source_modules` scoped only to `noeta.agent.backend` (`allow_indirect_imports=True`, so the backend reaching runtime indirectly through `noeta.sdk` is allowed; only direct imports of runtime internals are forbidden). `core-uses-only-protocols`, `runtime-no-providers`, `providers-only-protocols` are unchanged (provider-neutral, see `docs/adr/provider-neutral.md`).
