# Workspace is a first-class entity; a session pins one absolute path stored directly; permission collapses to three modes, switchable per turn

## Context

One process must serve multiple working directories and multiple providers at once, with each session choosing its own run configuration; at the same time a session must be able to reliably resume back to its original working directory, even if that directory is later renamed or deregistered. An early scheme treated a workspace as a "named subdirectory under a base pool," stored only the name in durable, and refused to let the frontend pass an absolute path directly; later, to "actually work inside a real project directory," it added symlink + `workspace_path` glue. That glue violated its own decision (a symlink does exactly the "frontend passes an arbitrary path" thing that was explicitly refused), and it made resume secretly depend on a stray symlink that is not in the event log and whose name carries an irreversible hash â€” once the pool is cleaned up or migrated, resume can't find the directory or runs against the wrong one. This decision overturns that glue, promotes workspace to a first-class entity, and pins the session's working-directory model to "one session pins one absolute path, stored directly into durable."

Layering follows `library-sdk-architecture.md`: allocation and management in the agent layer, execution and contract in the SDK + runtime layer; catalog neutrality follows `provider-neutral.md`.

## Decision

**Permission collapses to three modes, and the entire plan path is torn out by the root.** The permission mode keeps only `default` / `acceptEdits` / `bypassPermissions`. The `enter/exit_plan_mode` tool, `_maybe_plan_mode_decision`, `plan_mode_enabled` wiring, `AgentSpec.Capabilities.plan_mode`, the `plan_mode_read_only` guard, `GuardContext.task_phase`, and the entire "plan-phase state machine" are all deleted â€” pure subtraction.

**permission_mode sinks to the session level, switchable per turn, not persisted.** The frontend sends the currently-selected mode each turn, the backend uses the received value, and if none is received falls back to the host default; a per-turn mode overrides the host's explicit `require_approval_tools`. **Not persisted**: resume does not re-run the permission gate (recorded approvals/denials are read back directly from the log), so the permission mode value passed at resume never affects the outcome â€” it is **not written to the event log**, and a process-local transient carrier (`note_turn_permission`) hands the value across the async boundary.

**provider sinks to the session level; the registry is in the agent layer; the catalog stays neutral; (provider, model) folds into the model binding.** The catalog still manages only "model spec / price / capability," carrying no provider shape; the agent layer gains a provider registry (`(adapter, base_url, key, model list)`), and the runtime's `SdkHost.provider` (a single instance) becomes `SdkHost.providers` (a nameâ†’instance table). The frontend passes the provider's **name** (never a secret), and the backend validates `(provider, model)`. The provider **folds into the existing model binding** as a pair (reusing the ModelBound fold path, without opening a separate `ProviderBound` event); durable records only the name (the instance carries secret/connection and cannot enter the event log).

**skills merge across three layers, memory is a single global layer.** When workspace sinks, the fs root follows, but skills and memory do **not** drift with the per-session workspace. skills: built-in < global `~/.noeta/skills` < workspace-local `.noeta/skills`, all merged, with workspace winning. memory: pinned to one global directory (default `~/.noeta/memories`), independent of workspace.

**workspace is promoted to a first-class entity, landing as a lightweight registry in the agent layer.** `Workspace = {id, display name, absolute path}`. The registry lands in the **agent layer, not the runtime's event log**, backed by a JSON file (default `~/.noeta/workspaces.json`); it adds `POST/GET/DELETE /workspaces` (`DELETE` removes the registry entry, not the disk directory).

**A session pins the full path, durable stores the absolute path, and resume reads only the path, never the registry.** `TaskHostBoundPayload` pins `workspace_dir` (storing the resolved absolute path). When creating a task, it queries the registry once for the path, expands it on the spot, and pins it into durable; resume reads this absolute path directly and never touches the registry (even if the workspace is renamed or deleted, an old session still resumes back to its original directory). The engine cache key's workspace dimension changes from "name" to "full path."

**Path authorization is opened up, with zero allowlist.** When creating a workspace, the client may pass any absolute path, and the backend only validates "it is an absolute path, and a creatable/existing directory" â€” no directory allowlist.

**A bare session uses a configured default workspace (a "default" bucket).** When `POST /tasks` carries no `workspace_id`, the session lands in the configured default workspace, rendered into the same bucket.

**Delete symlink / `workspace_path` / name resolution / slash sniffing â€” pure subtraction.** The `workspace_path` + symlink glue previously added for "working inside a real project directory" is torn out by the root.

**A clean break, no historical compatibility.** The old single-provider environment variable and the old "by name" `TaskHostBound` are given no backward compatibility â€” single-user, dev-period tooling, resettable.

### Follow-up layout adjustment (2026-06-28): rebuilding session-level workspace/model/effort on the post-restructure layers

`runtime-sdk-app-restructure.md` later collapsed the multi-workspace **agent layer** this decision had built up (the set with `workspace_registry` / `workspace_resolver` / `/workspaces` endpoints / a provider registry) into a **single-workspace thin product backend**: `noeta.agent.backend.EngineRoom` wraps a `noeta.sdk.Client` pinned to a single `workspace_dir` / single provider / single model, and import-linter restricts the backend to importing only `noeta.sdk`. The underlying **runtime** machinery survives as-is (`SdkHost` still keeps both workspace and provider as engine-cache dimensions; the driver still welds `TaskHostBound.workspace_dir` and folds/resolves it on resume); what was narrowed is only the **agent-layer wiring + that thin Client surface**. The following rebuild, on the new topology, the session-level workspace/model/effort that the codex-style frontend needs:

- **The workspace registry + `/workspaces` CRUD move into `noeta.agent.backend`** (the post-restructure "agent layer"), still the same `~/.noeta/workspaces.json` and the same zero-allowlist path validation, with the registry now landing in `noeta.agent.backend.workspace_service`.
- **The thin `noeta.sdk.Client` surface is widened to re-expose what the restructure collapsed**: `Client.start` forwards a per-session `workspace_dir`; `Client.start` / `send_goal` forward a per-turn `effort`; `Client.__init__` takes `allowed_models` â†’ the driver's `model_allowlist`. These all forward into parameters that **already exist** on `InteractionDriver` / `SdkHost`, with the engine untouched. Reasoning: after the restructure the backend can only import `noeta.sdk`, so the Client is the only seam; the restructure originally collapsed it to a single workspace as a simplification for a thin backend that had no frontend at the time, and this restores it. **Zero mapping**: workspace is passed once at `start`, welded into durable's `TaskHostBound`, then folded/resolved each turn â€” no Client pool, no per-execution rebuild, no taskâ†’workspace table.
- **Local deployment: the config's `models` list = the model allowlist.** noeta-agent runs as `LOCAL_PRINCIPAL` (âŠ¤, no trust boundary), so `web-task-creation.md`'s `selector âˆˆ principal.allowed_models âˆ© deployment-allowlist` specializes here to "the deployment config's `BackendConfig.models`," passed in as `allowed_models` through the widened Client. This lets a real model id (like `gpt-5.5`) pass `_authorize_selector` without per-principal configuration â€” the same single-user trust assumption as the zero-path allowlist / the client self-elevating to bypassPermissions, **not** a cross-deployment security fallback, and it should be recorded at the seam as a local-âŠ¤ specialization.
- **provider is still a single one in v1.** The per-session provider registry above is built in the runtime, but is **not exposed to the frontend here** (the codex screenshot switches model + effort, not provider). The widened Client keeps a single provider, but must not be hard-coded in a way that can't be extended â€” when multi-provider is needed, wire the providers-registry half back out.

## Rationale

- **Sinking the three run-config pieces from "pinned at process start" down to the session level** is so one process can serve multiple workspaces / multiple providers, each session choosing its own; layering follows library-sdk (management in the agent layer, execution in the SDK layer).
- **Tearing out the entire plan**: it drags along a "plan-phase state machine" cutting across SDKâ†’engineâ†’guard, a vertical feature far heavier than a single dropdown, and cutting it also slims the engine. Deleting only the dropdown and keeping the machine = the entry is blocked but the machine still spins, pure clutter.
- **provider folds into the model binding rather than opening a new event**: provider and model are a pair to begin with, selected and switched together, so the existing fold path is reused; durable records only the name because the instance carries secret/connection and cannot enter the event log.
- **memory a single global layer**: memory is written, and splitting it in two forces an extra cut of "which layer does new memory land in"; besides, memory's value is "remembering across contexts," and slicing it by workspace only weakens it.
- **Overturning symlink/named-subdirectory**, because that set was "glue that violates its own decision": the symlink does exactly what 0036 explicitly refused (the frontend passing an arbitrary path directly), preserving only the appearance that "durable stores a name" while losing the security boundary; and resume secretly depends on a stray symlink that is not in the event log and whose name carries a one-way irreversible hash â€” once the pool is cleaned/migrated, resume can't find the directory or runs against the wrong one. In one line, the core model: **each session pins one working directory (absolute path, into durable, resume reads only it)**; workspace is merely a product-layer convenience for "letting multiple sessions share a directory + grouping," kept off the resume critical path.
- **The zero path allowlist is a deliberate trade-off**: single-user, locally trusted; "the frontend passing `/Users/leo/Documents/noeta`" and "the client self-elevating to `bypassPermissions`" are the same trust assumption; fine-grained directory authorization is deferred until multi-tenancy (this is the first hole to come back and patch later).
- **Bare sessions sharing a default workspace is a revision of the early "each bare session mkdirs its own `session-<uuid>` scratch directory."** codex's session panel groups by workspace, and the early set would make each bare session a singleton group of one conversation, so grouping loses meaning; under this decision's revision, bare sessions share the configured default workspace, rendered into one bucket, with per-session scratch directories deferred.

## Alternatives considered

1. **A new `SdkHost` per session.** Rejected: it would duplicate heavy machinery like provider/registry; changed to passing workspace once at `start`, welding it into durable, and folding/resolving each turn.
2. **The frontend passes an arbitrary absolute path directly (the early stance was to refuse).** The early refusal was on the grounds of "exposing the whole filesystem"; this decision accepts it in the single-tenant context, because this was already secretly being violated by the symlink, and it is the same trust assumption as bypassPermissions.
3. **Open a separate `ProviderBound` event / nail the providerâ†’model mapping into the neutral catalog.** Rejected: provider and model are a pair to begin with and should reuse the fold path; nailing it into the catalog would break catalog neutrality.
4. **Split memory into two layers (project-level + global).** Rejected: it forces an extra cut of "which layer new memory lands in," and slicing by workspace weakens memory's value.
5. **Durable-ized per-turn permission (needs a new event type) / setting permission only at task creation (changing it requires a new session).** Rejected: permission replay doesn't affect the outcome and needn't enter the log / the latter has poor UX.
6. **Resume depending on a stray symlink (an early form, deleted).** Rejected: the symlink is not in the event log and can't be rebuilt from persistent data; changed to querying the registry once at task creation and then freezing it into durable.
7. **All bare sessions share a default workspace (once rejected early).** The early rejection was "it would leave each group in the session panel with only one conversation, so grouping loses meaning"; after the codex frontend introduced grouping by workspace, this reason inverts to support bare sessions sharing a default workspace â€” otherwise "a scratch directory per bare session" is what would make every group a singleton.

## Consequences

- Landings are scattered across several places: `noeta.client.host` (the `providers` table, `_workspace_dir_for` with name resolution removed, the engine cache key dimension switched to full path), `noeta.execution.driver` / `noeta.execution.resolver` (binding fold, writing the full path), `noeta.execution.skills` / `noeta.execution.memory` (multi-directory / global), `noeta.policies` (plan deletion).
- Protocol side: `noeta.protocols.events` (`TaskHostBoundPayload.workspace_dir`, the model-binding fold carrying provider); on the runtime side plan and task_phase are deleted.
- Agent layer: the workspace registry and CRUD endpoints now land in `noeta.agent.backend` (`workspace_service` etc.), and the equivalent under the early agent host has been superseded; `EngineRoom` wraps a Client pinned to a single workspace, and per-turn permission is handed across the async boundary by the process-local transient carrier `note_turn_permission`.
- Costs and follow-up notes: the zero path allowlist is a deliberate opening under the single-user trust assumption, and fine-grained directory authorization must be revisited at multi-tenancy; per-session scratch directories and multi-provider frontend exposure are both deferred; the clean break means old recordings/old environments get no compatibility.
