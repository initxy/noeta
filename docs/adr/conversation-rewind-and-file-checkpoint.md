# Conversation rewind + AI-edited file checkpoints: a marker event reusing snapshots + per-turn baselines at the edit choke point

## Context

Undoing a user message means rewinding that message, its AI output, and every file changed in that turn (including subtasks) together. Two invariants constrain this: **the event log is append-only and immutable** (see event-sourced-truth.md), and **fold/resume rebuilds state purely from the event log and never touches disk** (see replay-verify-tolerance.md). Guiding principle: **reuse, don't build new**.

## Decision

- **Scope = a true rewind (conversation + files together, including subtask changes); a single "undo" action.** Undo is anchored to a user message bubble = rewind to just before that message was sent. Just one action (no choice of "code only / conversation only / both"). Undoing a message in the middle rewinds it and all subsequent turns together. **Non-goals**: shell / external changes are not rewound; user manual edits and concurrent-session edits are not tracked; there is no transactional atomic rollback; there is no cherry-pick-style rewind (preserving later turns).
- **Conversation rewind = appending a `TaskRewound` marker event; fold treats it as "an old snapshot overlaid on the current state."** No truncation, no forking. On rewind, fold the state to the target `seq N`, serialize it, put it into the ContentStore to get a `state_ref`, then append `TaskRewound{target_seq, state_ref}`. **The only change to fold**: instead of "pick the latest snapshot as the baseline," pick "whichever has the highest seq among the latest snapshot and the TaskRewound." The N+1..M segment becomes dead history (still in the log, still auditable); append-only is intact.
- **File snapshot backend = ContentStore content-addressing + dedup; one checkpoint = a manifest of `{path → content ref}`.** No shadow git repo, no whole-tree hard copy; file content is stored in the ContentStore by fingerprint (identical content stored once), and the manifest itself is stored there too, so the whole checkpoint collapses into a single ref.
- **Coverage = only files the AI directly edited / wrote, captured at the sole `handle_tool_calls → ToolRuntime.invoke` choke point; shell / external changes are explicitly excluded.** Files changed indirectly by an AI-run script (`npm`, `mv`, etc.), as well as user / concurrent edits, are not tracked. Upside: no directory scan, no .gitignore filtering, and `node_modules` never enters a snapshot.
- **File state = a fold projection of the event log; an edit event also records an "edit-before content" ref; writing to disk is a live-only side effect that fold/resume skips.** Every edit event records the full "content before edit" as a ContentStore ref (nearly free after dedup), so file state can be folded from edit events just like conversation state. It is **not** an out-of-log side store—the baseline ref is inside the event, and fold can reproduce the same checkpoint.
- **Baselines are stored per "turn" (not per edit), stored at first touch, and recovery takes "the first baseline after the landing point."** When a file is edited for the first time within a turn, store its "before edit" content as that turn's baseline; subsequent edits in the same turn store nothing more. Recovering to turn T = for each file touched in turn ≥ T, write back its first baseline at or after T; if the baseline is "did not exist" (created by the AI), delete the file. The table tracking "has this file been stored this turn" is a **runtime accelerator** (mirroring background shell's `ProcessRegistry`), cleared each turn and never persisted; the authoritative data is the baseline ref on the edit event.
- **Three knobs cap the data volume**: a per-turn storage cap + a per-file size limit (over the limit or binary → skipped, not entering the checkpoint) + GC of dead history / dead blobs (hung off the existing compaction; v1 runs without it too).
- **Subtask cascade: a shared per-turn baseline gate + descendant enumeration; reuse the parent/child graph rather than the cancel-cascade traversal.** Subtasks share the parent's same working directory (see workspace-and-session-path.md), their edits land on the same disk, and a rewind must undo them together. Capture is free (subtask edits go through the same choke point); **the only change—the per-turn baseline gate goes from "one per task" to "one shared across the whole delegation tree"** (otherwise, after the parent edits X, a subtask would store the mid-turn dirty X as a second baseline). Because in v1 subtasks run in-process, serially, one lease at a time, the shared gate is just passing a reference—no lock needed. Recovery enumeration (the only newly-written traversal): the parent stream frames the seq range of turn T → collect `SubtaskSpawned` → recurse into each subtask stream to assemble the whole descendant tree → take the union of turn T's parent baselines and all descendant streams' baselines and recover.
- **UI: a user bubble's "undo" → confirmation prompt → one-click full rewind; the message text is refilled into the input box, the message disappears, and any in-flight turn is cancelled before rewinding.**

## Rationale

- **The file checkpoint is genuinely new, not an extension of the existing snapshot**: today's snapshot captures only the four in-memory state slices of a Task (to speed up fold), and **never touches the file system**; file changes are the side effect that fold/resume deliberately skips, so nothing in the event log can rebuild the file system.
- **A marker event, not truncation / forking**: truncation directly shreds append-only (breaking audit, dangling external refs, history no longer foldable); forking keeps append-only but forces you to maintain "one logical task = a tree of flows," which both the read model and the frontend would have to understand—very heavy. A marker event gives the same "go back + change direction" experience, but is far simpler. `TaskRewound` is recorded, and fold just reads it and skips the same segment by the same rule.
- **ContentStore dedup gives you near-free whole-tree fidelity, paying only for the changed files**—exactly git's internal mechanism, but without introducing git (no fighting the user's `.git`, no second VCS).
- **Capturing only direct AI edits (same limit as Claude Code)**: no scanning / filtering needed; the cost is that files generated by an AI-run script cannot be rewound, which is fine for a single-machine personal local tool.
- **Edits record an "edit-before content" ref, and file state is a fold projection**: not a ProcessRegistry-style out-of-log side store—that would make fold/resume unable to reproduce the checkpoint. The actual write to disk is a live-only side effect that fold/resume skips (just like the edit itself).
- **Per-turn storage, not per-edit**: the rewind landing point is a turn boundary, so only a per-turn baseline is needed; per-edit would double the data.
- **The shared subtask gate**: v1's serial execution is a clean precondition—if it ever goes truly parallel, the shared gate needs a lock, and "one baseline per file" must be redesigned for concurrent first-touch (noted). Reuse the parent/child graph rather than the cancel-cascade traversal, because the latter only scans in-flight, undriven subtasks, whereas a rewind targets already-terminal descendants.

## Alternatives considered

1. **Truncate the event log (physically delete after N).** Rejected: shreds append-only—audit breaks, external refs dangle, history is no longer foldable.
2. **Fork the event stream (git-style branching).** Rejected: keeps append-only but forces maintaining a tree of flows that both the read model and the frontend must understand—very heavy.
3. **A shadow git repo for file snapshots.** Rejected: introduces a second VCS, fights the user's `.git`, adds an external dependency; ContentStore dedup already matches git's blob model.
4. **A whole-tree hard copy / whole-tree scan to capture shell changes.** Rejected: copying a large project blows up disk; scanning needs .gitignore filtering + a size limit—overengineering for a single machine. Changed to capturing only direct AI edits.
5. **File snapshots as an out-of-log side store (ProcessRegistry-style).** Rejected: fold/resume can't reproduce the checkpoint.
6. **Store baselines per-edit.** Rejected: doubles the data.
7. **Offer a "code only / conversation only / both" choice.** Rejected: what the user wants is a one-click full rewind.
8. **Reuse the cancel-cascade traversal for subtask enumeration / centralize subtask baselines into the root task flow.** Rejected: the former only covers in-flight, undriven subtasks; the latter introduces cross-flow writes on the edit hot path. Changed to each flow recording its own baselines, enumerating descendants at recovery and merging.

## Consequences

- The per-turn baseline gate, file-state fold + recovery, and live-only write-back / deletion land in `noeta.runtime.file_checkpoint`; the fold rule "pick whichever of snapshot and TaskRewound has the higher seq as the baseline" lands in `noeta.core.fold`; `TaskRewound` and the edit events carrying an "edit-before content" ref land in `noeta.protocols.events`.
- Passing the shared gate at subtask Engine construction, capture at the `ToolRuntime.invoke` choke point, and the recursive `SubtaskSpawned` descendant enumeration land in `noeta.execution`.
- The `/tasks/{id}/rewind` endpoint, cancelling the in-flight top-level turn before rewinding, and triggering the recovery write-to-disk land in the backend host layer.
- Follow-up note: v1's serial subtask execution is the precondition for the shared gate being lock-free; going truly parallel requires reworking the shared gate and "one baseline per file"; shell / external changes and user manual edits are always outside rewind coverage.
