# Multi-host Postgres deployments fence lease-checked writes without an epoch column: atomic in-tx row-share check closes the zombie window, and DB-clocked expiry + global-advisory serialisation closes the distributed-timer and skew gaps

## Context

Single-host multi-worker concurrency (a resident worker pool sharing one host process, with `release_yield` handing seeded leases to that pool) is in place. **Multi-host** is the next capability: several host processes each running a worker pool, all sharing one Postgres database. Sqlite and in-memory adapters stay single-host by definition — no cross-process sharing is supported there — so this decision is scoped to the Postgres adapter only.

The existing lease model (worker-lease-model.md) plus the durable wake invariants D1–D6 (subtask-fanout-and-durable-wake.md) already guarantee at-least-once delivery + idempotent consumption on a single host. Step-attempt recovery (step-attempt-recovery.md) explicitly anticipates fencing: "recovery always runs under a lease and the seal is a lease-checked append, so a fencing epoch that fences leases automatically fences recovery too." That sentence is the seam this ADR locks into.

Two correctness gaps appear when moving from one host to many, both Postgres-specific:

1. **Zombie-append window (G1).** `PostgresEventLog._append` validates `lease_id` by calling `lease_validator.is_lease_valid(task_id, lease_id)` on the dispatcher's **separate read-only connection**, outside the emit transaction's locks (postgres/eventlog.py:258-268; postgres/dispatcher.py:827-838). Between the "valid" answer and the subsequent `INSERT INTO events ... COMMIT`, another host's `requeue_stale()` can reclaim the lease and hand it to a new worker. The original (zombie) worker's write then lands on the stream **after** the new generation has started, violating single-writer-per-lease and the ordering guarantee step-attempt recovery depends on. The window can be arbitrarily long (GC pause, SIGSTOP, scheduler preemption); it does not depend on clock skew.

2. **Clock skew (G2).** `lease_expires_at` is computed client-side by `self._now()` (default `time.time`, postgres/dispatcher.py:107), and `is_lease_valid` / `requeue_stale` / `fire_due_timers` compare against per-host clocks (postgres/dispatcher.py:691,838; postgres/dispatcher.py:746). With skew, host A can think a lease is still live while host B's `requeue_stale` thinks it expired, producing divergent lease-currency views. `TimerFired.fire_at` is an Engine-computed wall-clock deadline recorded into `wake_on_canonical` — the deadline bytes themselves cannot change, but the "is it due?" comparison is where skew bites.

A third concern is often raised independently — distributed timer sweeps producing double-fire or missed fires — but the existing design already contains the right locking shape; this ADR must argue that it scales across hosts (see "Distributed timer due-check" below).

## Decision

Scope this to Postgres only. No epoch column. Three targeted changes:

**D1. Atomic in-transaction lease check for EventLog appends (closes G1).** Precede the cross-connection `is_lease_valid` call inside `PostgresEventLog._append` with an **in-tx SELECT ... FOR SHARE** against `dispatcher_tasks` that checks `lease_id` equality, `status='leased'`, and non-expiry against the database clock — all inside the same BEGIN/COMMIT that holds the per-stream advisory lock and does the INSERT. Concretely, when `require_lease and lease_id is not None` **and a lease validator is bound** (no validator bound → no check at all, exactly as today), `_append` issues:

```sql
SELECT 1 FROM dispatcher_tasks
 WHERE task_id = %s AND lease_id = %s AND status = 'leased'
   AND lease_expires_at > EXTRACT(EPOCH FROM clock_timestamp())::double precision
   FOR SHARE
```

inside the already-open emit transaction, at the **same position the validator call sits today** (after the idempotency dedup and the `expected_seq` check, before the INSERT) — so the existing error-precedence and dedup-before-validation semantics the contract suite pins are unchanged. A returned row proves the lease is current in this database, and the `FOR SHARE` row-share lock blocks any concurrent `UPDATE` of the dispatcher row (which is what `lease` / `release` / `requeue_stale` / `fail` / `heartbeat-cap` do when they change the lease) until this emit transaction commits or rolls back; the registry is not consulted. Zero rows → **fall back to the bound registry's `is_lease_valid`** (the Protocol seam): a same-database wiring (the production stack) answers False there too and raises `InvalidLease` exactly as today, while a mixed wiring — e.g. an `InMemoryDispatcher` validating a `PostgresEventLog`, which the contract suite exercises — keeps its exact current semantics because the lease lives in the registry, not in this database's `dispatcher_tasks` table. Mixed wirings get no atomic fence (the resource and the lock are not in one database — that is Kleppmann's boundary, see "Why no epoch?"), and they are not a production shape.

The Python-side `LeaseRegistry.is_lease_valid` Protocol method is **kept** for the in-memory and sqlite backends and for any external tooling; the Postgres EventLog consults it only when the in-tx probe finds no row. The Protocol-level seam stays — other backends continue to satisfy it exactly as today.

**D2. Database clock for lease-expiry comparisons (closes G2).** When `now` is **not injected** (the production default), Postgres dispatcher expiry computations — the `lease_expires_at` written by `lease()` and `heartbeat()`, and the expiry tests in `is_lease_valid` / `requeue_stale` / the new D1 check — use `EXTRACT(EPOCH FROM clock_timestamp())::double precision` (Postgres' always-advancing transaction-clock function, not the statement-start `now()`) instead of `self._now()`. The `fire_at` deadline on `TimerFired` stays a client-supplied wall-clock value (it is a recorded payload byte); the due comparison in `fire_due_timers` continues to compare `fire_at <= now-caller-parameter` so it shares the Engine's wall-clock base — but for the postgres adapter specifically, the read-probe and write sweep use `EXTRACT(EPOCH FROM clock_timestamp())::double precision` when `now` was not injected. When `now` **is** injected (tests), every comparison stays exactly as today, preserving the deterministic-clock test seam.

The contract tests already pin expiry semantics through the injected-`now` fixture path; they continue to pass unchanged. New contract tests (see spec) exercise the multi-host paths using two adapter instances sharing one DSN.

**D3. Record `worker_id` on the dispatcher row.** The current code does `del worker_id` in `lease()` (postgres/dispatcher.py:283; sqlite equivalent). Add a nullable `worker_id TEXT` column to `dispatcher_tasks` (migration 3, postgres only — sqlite/in-memory are single-host by definition and defer), populated by `lease()` and cleared on every state transition that drops `lease_id`. This gives audit/observability (which host holds which lease) and is load-bearing for diagnosing cross-host races — it is not a fencing token.

### Why no epoch?

Every protected write in the system either (a) goes through the Postgres EventLog (now fenced atomically by D1), (b) is a dispatcher lifecycle mutation already serialised by the global dispatcher advisory lock (`pg_advisory_xact_lock(_ADVISORY_CLASS_DISPATCHER, 0)`, postgres/dispatcher.py:136), or (c) goes to the ContentStore, which is content-addressed and immutable (put is idempotent first-write-wins on the content hash, postgres/contentstore.py:58-64). There is no write to a resource outside the shared Postgres database that depends on lease currency for correctness. Kleppmann's fencing-token argument is required precisely when the protected resource cannot be brought into the same atomic transaction as the lock check — e.g. a blob store that does not support row-share predicates, or an external API. Here the resource (events table) sits in the same database as the lock (dispatcher_tasks row), so a row-share lock gives strictly stronger semantics (it holds off writers until commit) than a compare-and-set on a token (which catches stale writers after the fact but allows the stale write to reach COMMIT).

A monotonic epoch would still work — every lease-grant would bump it, every write path would compare `WHERE lease_epoch = <expected>` — but it adds a column, adds comparison logic to a dozen write paths, and still requires an in-tx check on the EventLog append path to be correct (an out-of-tx epoch check has the same zombie window H1 identifies). D1's row-share lock is exactly as correct with strictly less code and no schema churn beyond the optional worker_id audit column.

### Per-transition fence argument

Each dispatcher lifecycle transition is reviewed below for zombie windows in the multi-host world.

**lease()** (postgres/dispatcher.py:261-329). Runs under the global dispatcher advisory lock. `SELECT` the FIFO ready task, then `UPDATE dispatcher_tasks SET status='leased', lease_id=<new uuid>, lease_expires_at=<deadline> WHERE task_id=<id>`. The UPDATE is atomic with the lock hold; no other host can lease the same task (the FIFO selection + UPDATE is one serialised transaction; the `ix_dispatcher_ready` partial index makes the SELECT deterministic). The previous `lease_id` value is overwritten, so any in-flight emit from the prior generation is either (a) already committed before this transaction's SELECT (so it is a valid write from the prior lease's era, sequenced before this lease's writes), or (b) still in its emit transaction holding the row-share lock, in which case our UPDATE waits for it. In practice `lease()` never queues behind a probe at all: it only UPDATEs a row it just selected with `status='ready'`, the fence probe only matches (and locks) `status='leased'` rows, and a probed row cannot change status while the probe's lock is held — so any lock the probe took is released (at its COMMIT) before the row can ever become ready. The full wait-for analysis lives in the implementation spec §"Lock ordering (deadlock freedom argument)".

**heartbeat()** (postgres/dispatcher.py:331-393). Under the global dispatcher lock; fetches by `lease_id` equality on the unique index `ix_dispatcher_lease_id`. If the cap is hit, the same transaction force-releases (sets status='ready' or 'suspended', clears lease_id). A zombie whose heartbeat hits after reclaim raises `InvalidLease` (row-by-lease_id returns no row) — path already correct. Success path extends `lease_expires_at`; this is a same-lease write, so it cannot accidentally fence another generation.

**release()** (postgres/dispatcher.py:395-519). Under the global lock; `_fetch_task_by_lease(lease_id)` returns None → `InvalidLease`. Every branch clears `lease_id` and sets the new status atomically. D1's FOR SHARE on an in-flight emit holds the row-share lock until commit; if a release commits first (clearing lease_id), the emit's FOR SHARE finds zero rows and raises `InvalidLease` — exactly the desired zombie rejection.

**release_yield()** (postgres/dispatcher.py:521-557). Same shape as release: clears `lease_id`, sets status='ready' with a fresh `ready_order`, preserves `matched_wake_event_canonical`. Zombie rejection identical.

**fail()** (postgres/dispatcher.py:559-624). Under the global lock; same `_fetch_task_by_lease` gate. Retryable path goes back to ready, non-retryable goes to terminal. Same zombie-rejection property.

**wake()** (postgres/dispatcher.py:626-681). Under the global lock. For a suspended task whose `wake_on` matches, flips to ready and installs a `matched_wake_event_canonical`. Wake does not require the holder's lease_id (any producer can wake a suspended task — this is the protocol-level wake channel). This is correct: wake is a lease-**creation** event (it produces a ready task that some future lease will pick up), not a lease-protected write. A concurrent emit from the still-leased worker (e.g. the wake arrives "early" while the worker is about to suspend) races with the suspend-side release — the release is the one that decides whether the wake was early (buffered) or late (matched); the existing drain logic inside release handles this correctly (postgres/dispatcher.py:500-513) and is single-host logic serialised by the global lock.

**requeue_stale()** (postgres/dispatcher.py:683-744). Under the global lock. Selects all rows with `status='leased' AND lease_expires_at <= <now>`, then either (a) increments `reclaim_count` and transitions to ready, or (b) at `reclaim_max` drops to terminal with `stale_reclaim_exceeded`. This is the **reclaim path** that hands a new lease to a new worker. D1's FOR SHARE guarantees that any in-flight emit from the reclaimed zombie either commits BEFORE the requeue_stale UPDATE (in which case it is a pre-reclaim valid write and the new lease's writes follow it in seq order), or blocks on the FOR SHARE until requeue_stale commits (which clears lease_id, so the FOR SHARE finds zero rows and raises `InvalidLease` in the zombie). There is no window in which the zombie's write lands AFTER requeue_stale has committed and a new lease has been minted — exactly the guarantee the recovery argument needs.

**fire_due_timers()** (postgres/dispatcher.py:746-811). Under the global lock. Selects suspended rows with `fire_at <= <now>` and flips each to ready, installing the canonical `TimerFired(fire_at=<stored>)` as the matched wake. This is a producer-side wake, same category as `wake()`: it creates ready tasks, does not require a lease_id. For multi-host correctness see "Distributed timer due-check" below.

**enqueue()** (postgres/dispatcher.py:203-259). Under the global lock. Existing three-path behaviour: no row → insert ready; ready → no-op (FIFO preserved); non-ready (suspended/terminal/**leased**) → force-clear every state column including lease_id and matched_wake_event_canonical, set status=ready with a fresh ready_order. This last branch — enqueue on a currently-leased task — is a **lease-preempting write**: any host calling enqueue on a live task unconditionally strips the lease. The contract suite pins the ready/terminal/suspended paths (test_enqueue_already_ready_is_idempotent_no_op, test_enqueue_existing_terminal_task_resets_to_ready, test_enqueue_existing_suspended_task_resets_to_ready) but does **not** currently pin enqueue-on-leased behaviour — contract tests never enqueue a task while holding its lease (tests/test_dispatcher_contract.py:112-178). Call-site review shows the four non-test enqueue callers are:

- `ChildLifecycleObserver._on_task_created` (observers.py:155) — enqueues a freshly-created child task immediately after the `TaskCreated` event. The child was just created; it cannot be leased yet (no one has had a chance to lease it, because the lease flow requires the task to be in `ready`). This site is safe.
- `BackgroundSubagentRegistry._submit` (background_subagent.py:138) — same pattern for background children. Safe.
- `InteractionDriver._seed_start` (driver.py:646) — enqueues the just-created root task immediately before a targeted `lease(task_id=...)`. No other host could have leased the task because the seed path runs task-creation + enqueue + targeted-lease in a sequence where the task_id is freshly minted and not yet observable to other workers (the worker pool picks from the global ready queue; the targeted lease claims it immediately on this host). Same-host concurrency: the enqueue + targeted lease are not in one DB transaction today, but they run synchronously on the caller thread before release_yield. **Multi-host caveat**: between enqueue and targeted lease, another host's FIFO poll could steal the task. This is a bug latent in the current code that would manifest when workers run on multiple hosts; it is **not** introduced by fencing. It needs an implementation decision from the maintainer (see implementation spec §"Out-of-scope / maintainer decisions"). The enqueue itself remains a legal lease-preempting write; the fencing design does not change its semantics.
- `subtask_drain._descend_to_parent_wake` (subtask_drain.py:304) — enqueues the parent after releasing the parent's temporary lease in the "parent had no wake" branch. The parent's lease was just released by THIS host, so the parent is not in 'leased' state. Safe.

So enqueue-on-leased is reachable in principle through the public API (a caller could race a targeted enqueue against another host's lease) but is not exercised by any current in-tree caller against a live lease. The current "force-strip" semantics are preserved as-is; tightening them (e.g. raising on leased) is explicitly out of scope for this round and flagged for maintainer decision.

**restore_task / purge_task** (postgres/dispatcher.py:880-991). These are adapter-local maintenance seams (rewind and delete-session), not on the Dispatcher Protocol. Both run under the global lock and wipe or overwrite dispatcher row state unconditionally. They are operator-initiated actions, not worker writes; fencing does not constrain them.

**Idempotency table** (events `idempotency`). Keyed on `(task_id, lease_id, idempotency_key)` (migrations.py:89-96; eventlog.py:228-240,292-303). Entries are written by the SAME emit transaction that writes the event row, under the same per-stream advisory lock and (with D1) the same FOR SHARE on the dispatcher row — so an idempotency entry is only created when the lease checks out atomically. A replayed emit with the same `(task_id, lease_id, idempotency_key)` returns the cached seq even across hosts; this is correct because the idempotency key is a UUID generated per emit call, so a cross-host retry cannot accidentally collide. After a reclaim, the new lease has a new `lease_id`, so the new worker's emits get a fresh idempotency namespace.

**ContentStore** (postgres/contentstore.py:48-65). `put(body, media_type)` does `INSERT ... ON CONFLICT (hash) DO NOTHING` — content-addressed, first-write-wins, no lease, no per-task identity. No fencing required; a zombie writer putting the same content bytes as the new worker is a no-op and produces the same ContentRef.

### Distributed timer due-check

`fire_due_timers` is the periodic sweep that converts overdue `wait_timer` suspends into ready tasks. On one host the existing design is correct; on multiple hosts every host runs its own poll (WorkerLoop.maybe_poll_timers, worker.py:1586-1618), and we must argue no double-fire and no missed fire.

- **No double fire.** The flip (select due rows → UPDATE status='ready' + set matched_wake_event_canonical + clear fire_at) runs inside ONE transaction that holds the global dispatcher advisory lock (postgres/dispatcher.py:770-807). Two hosts entering fire_due_timers at the same moment serialise on the advisory lock; the first to COMMIT moves the due rows to ready and clears their `fire_at`; the second host's SELECT (still inside its own locked transaction, started after the first committed) sees zero due rows because `fire_at IS NULL` after the flip. The matched wake event is the stored `wake_on_canonical` bytes (i.e. `TimerFired(fire_at=<deadline>)`, not a newly-minted value), so the delivery is byte-identical regardless of which host swept.

- **No missed fire.** Every host polls on its `timer_poll_interval` (default 1 s); the first host whose poll lands after the deadline wins the sweep. If a host crashes between the probe (which reads on `_read_conn`, postgres/dispatcher.py:762-768) and the write transaction, another host's next poll will run the sweep (the write transaction is what actually flips state; the read-probe is an empty-poll optimisation and races benignly — the code comment at postgres/dispatcher.py:758-760 already documents this). There is no host-assignment of timers; all hosts share one FIFO-ready queue after the flip.

- **Read-probe race is benign.** A host's read-probe can return "nothing due", then a timer becomes due, and then the host skips the write transaction this cycle. The next cycle (or another host's concurrent cycle) fires it. Bounded by poll interval.

### Clock base

Three distinct clocks are relevant, and the design keeps each in its lane:

1. **Per-stream event ordering** uses `seq`, an always-incrementing integer assigned inside the per-stream advisory lock. Clock skew cannot reorder events within a stream — only the commit order under the advisory lock determines seq. This is already true and unchanged.

2. **Lease expiry / heartbeat / stale-reclaim** (D2). Production default: use Postgres `EXTRACT(EPOCH FROM clock_timestamp())::double precision`. This is one clock (the database server's clock) visible to every host identically, so there is no per-host skew on the expiry comparison. Tests inject a deterministic `now` callable and keep the existing client-side comparison. The residual risk: Postgres wall-clock can jump (NTP step) — a backwards step extends leases (safe, just delays reclaim briefly); a forwards step can expire leases early (bounded by `lease_seconds` heartbeats — the next heartbeat re-extends using the new clock). This residual is documented and accepted; operators running multi-host should run NTP on the DB server as they would for any multi-host system.

3. **Timer deadlines (fire_at).** The Engine computes `fire_at = time.time() + wait_seconds` at suspend and records it into `TimerFired.fire_at`, which is persisted as bytes inside `wake_on_canonical` and mirrored onto the `fire_at` column. The due-check compares `fire_at <= <now-parameter>`, where `<now-parameter>` is supplied by the caller (WorkerLoop passes `self._now_fn()` which defaults to `time.time`, worker.py:1504). In production under D2, the Postgres adapter uses `clock_timestamp()` for the sweep, while other hosts' WorkerLoops pass their own `time.time()` — so **skew between a host's wall clock and the DB clock can cause a timer to fire up to skew seconds early or late**, but it cannot cause double-fire (locking prevents it) and it cannot cause missed fire (every host polls; the DB-side sweep on one correctly-clocked host will eventually fire it). The Engine's `fire_at` and the DB sweep do not need to agree exactly; NTP alignment of hosts to within O(1 s) is the operational assumption.

### Completion-order theorem (link to step-attempt-recovery)

For any task, define a lease **generation** to be the interval between one successful `lease()` (which assigns a new `lease_id = L_i` and sets `status='leased'`) and the next transition that clears `lease_id` (release / release_yield / fail / heartbeat-cap / requeue_stale / enqueue-force-clear / restore/purge). The theorem: **every event emitted by generation L_i has `seq <` every event emitted by generation L_{i+1} on the same stream.**

Proof sketch:

- Generation L_i ends at the COMMIT of a dispatcher transaction that clears `lease_id` on the row; generation L_{i+1} begins at the COMMIT of a dispatcher transaction that sets `lease_id = L_{i+1}`. Both transactions hold the global dispatcher advisory lock, so their COMMITs are totally ordered — there is a well-defined "last write under L_i" and "first write under L_{i+1}" in commit order.

- Every EventLog emit under L_i either (a) COMMITs before the generation-ending dispatcher COMMIT, or (b) is still in progress when the generation-ending dispatcher UPDATE runs. In case (b), the emit transaction holds a `FOR SHARE` row lock on the dispatcher row (D1); the dispatcher UPDATE (which takes a row-level write lock) blocks until the emit transaction COMMITs or ROLLBACKs. So in case (b) the emit's COMMIT is ordered BEFORE the generation-ending dispatcher COMMIT.

- Therefore every committed event under L_i has a commit order BEFORE the generation-ending dispatcher COMMIT.

- Generation L_{i+1}'s `lease()` runs a dispatcher transaction that COMMITs AFTER the generation-ending dispatcher COMMIT (global advisory lock serialisation). Every EventLog emit under L_{i+1} starts AFTER its `lease()` COMMIT (a worker only emits after receiving the Lease) and holds `FOR SHARE` on the row that is already locked by L_{i+1}'s lease_id, so its COMMIT is ordered AFTER the L_{i+1} lease COMMIT.

- Per-stream seq is assigned inside the per-stream advisory lock in COMMIT order, so commit order on a stream equals seq order. Hence seq(L_i writes) < seq(L_{i+1} writes).

This is exactly the property step-attempt-recovery.md line 49 relies on: the seal (a lease-checked append under the new lease) cannot interleave with writes from the prior generation. Recovery can safely fold at the attempt boundary without double-applying or skipping events.

### Wake invariants D1–D6 are preserved

- D1: lease() does not clear matched_wake_event_canonical. Unchanged (postgres/dispatcher.py:276-281).
- D2: release() clears matched only via consumed_wake_event, validated byte-for-byte. Unchanged (postgres/dispatcher.py:418-436).
- D3: requeue_stale() keeps matched_wake_event_canonical when transitioning to ready (it only clears lease fields and increments reclaim_count). **Unchanged** — and D1's FOR SHARE now makes it safe against a concurrent emit that might otherwise have clobbered the wake state.
- D4: first-consume vs re-delivery reconciliation in `_run_woken` is entirely Engine-side fold logic (worker.py:882-1025), independent of fencing.
- D5: heartbeat-cap and fail() never clear matched. Unchanged (postgres/dispatcher.py:347-373, 559-624).
- D6: release(consumed_wake_event=X) validates X = stored matched byte-for-byte, else raises WakeConsumeMismatch and rolls back. Unchanged (postgres/dispatcher.py:419-430).

**release_yield** keeps matched_wake_event_canonical (postgres/dispatcher.py:538-551 — the UPDATE leaves the column untouched), so a wake delivered while a seed-lease is held survives the yield and is delivered to the worker that picks it up. This is the property `test_release_yield_preserves_pending_wake` pins (tests/test_dispatcher_contract.py:600-634); D1 does not change it. Cross-host: if host A holds the seed lease, host B delivers a wake while the task is 'leased', B's `wake()` takes the global dispatcher lock, sees the task is 'leased' (not 'suspended'), buffers the wake into `dispatcher_pending_wakes` (correct — early-wake buffering path), COMMITs. Host A then calls `release_yield`, which goes leased→ready without a wake_on set (so no drain of the pending wake at yield time). A subsequent lease/release-suspend cycle drains the pending wake as usual. Correct.

### Red lines

- **EventLog envelope and payload bytes are unchanged.** No new event types, no new envelope fields, no payload drift. `EventEnvelope` field set stays identical; `idempotency` table schema stays identical (its primary key still starts with `lease_id`, but `lease_id` values are the same uuid4 shape, just checked differently).
- **Old recordings fold and replay byte-identically.** No event bytes are reinterpreted; the migration only adds an optional `worker_id` column to a dispatcher-internal table (not the events table).
- **Storage contract tests (`tests/test_dispatcher_contract.py`, `tests/test_event_log_contract.py`) are only added to, never modified.** Existing semantics (including the enqueue-on-non-ready force-clear) are preserved.
- **Sqlite and in-memory adapters keep their existing client-clock behaviour.** They are single-host by definition; no changes.
- **No new API verbs on the Dispatcher or EventLog Protocols.**

## Rationale

- **Atomic in-tx FOR SHARE is strictly stronger than an out-of-tx epoch check** for same-DB resources, because it blocks the conflicting writer rather than merely detecting the conflict post-hoc. Kleppmann's fencing-token argument is for resources that cannot participate in the lock holder's transaction; here the resource (events table) lives in the same database as the lock, so we can do better.
- **DB-clocked expiry eliminates clock skew on the lease-liveness question** without requiring operators to run clock-sync daemons with tight bounds. The only remaining skew is between Engine-computed `fire_at` deadlines and the DB clock, which is a bounded delivery-jitter question (timer fires up to skew early/late), not a correctness question.
- **Keeping the `LeaseRegistry` Protocol seam** leaves the backend-agnostic shape intact: in-memory and sqlite adapters continue to implement `is_lease_valid` for the cross-connection validator pattern, and the Postgres EventLog's in-tx SQL check is an adapter-internal optimisation that does not leak to the Protocol.
- **Worker ID as an audit column** is low-cost (one nullable TEXT column, no index) and pays for itself the first time an operator needs to answer "which host held this lease when it went stale." Not using it as a fencing token avoids overloading an observability field with correctness semantics.
- **Not changing `enqueue()` semantics** avoids breaking any latent caller that relies on force-ready, and keeps the change scoped to the fencing properties. The maintainer can tighten enqueue in a later round if desired; the contract tests do not currently pin behaviour on leased tasks.

## Alternatives considered

1. **Monotonic epoch / fencing token column.** Add a `lease_epoch BIGNOT NULL DEFAULT 0` column; bump it on every lease grant; require every lease-protected write to include `WHERE lease_epoch = <expected>` (or `SELECT ... WHERE lease_epoch = <expected> FOR SHARE`); reject mismatches with InvalidLease. Rejected because (a) D1's FOR SHARE on `lease_id` equality already gives this exact property with fewer tables and less plumbing — `lease_id` is already a unique, never-reused random value, so equality on it is a strictly stronger fence than equality on a counter; (b) an out-of-tx epoch check (the naive implementation) would leave the same zombie window H1 identifies, so the epoch design would still need D1's in-tx locking; (c) more schema churn for no additional safety. Retained as a fallback if a future write path falls outside the shared DB (e.g. an external blob store with no transaction participation) — at that point an epoch becomes load-bearing.

2. **Move all EventLog writes through the dispatcher lock** (i.e. require emit to take the global dispatcher advisory lock in addition to the per-stream lock). Rejected: it serialises every event append across every task in the system through one global lock, destroying write concurrency for a hot path (events per step can be dozens). The per-stream advisory lock already gives correct per-task serialisation; adding the global lock would be a major throughput regression for a problem D1 solves with one SELECT.

3. **Use Postgres `SELECT ... FOR UPDATE` instead of `FOR SHARE`** in D1. Rejected: FOR UPDATE takes a write lock on the dispatcher row, which would (a) conflict with itself if two appends on different leases for the same task somehow ran concurrently (they don't — the per-stream lock serialises them, so FOR SHARE is sufficient), and (b) be stronger than needed. FOR SHARE allows multiple concurrent readers (which is the steady state — only one writer per stream, but that writer holding a share lock does not block snapshot reads from e.g. introspection tools). In practice either works for correctness; FOR SHARE documents intent more precisely.

4. **Use `now()` (statement timestamp) instead of `clock_timestamp()` for D2.** `now()` is frozen at transaction start, which would let a long-running emit transaction see a stale expiry cutoff; `clock_timestamp()` advances during the transaction, which is the correct behaviour for "is this lease still live right now as I commit this write?"

5. **Make `fire_due_timers` a single designated leader (one host sweeps).** Rejected: (a) requires leader election, which is a new distributed-systems primitive and an availability hazard (if the leader dies, timers stall until election completes); (b) every-host-sweep with the global advisory lock gives the same exactly-once semantics with zero additional machinery and fails open — any host can sweep. The cost is one extra lock acquisition per poll per host, which is negligible (empty polls already skip the write transaction via the read-probe fast path).

6. **Tighten `enqueue()` to reject (or no-op) when the task is currently leased.** Rejected for this round because (a) the contract suite does not currently pin enqueue-on-leased behaviour either way, but the code's documented contract says "enqueue on any non-ready status transitions to ready," and changing that silently would be a behaviour change that the contract tests would not catch until a new test is added; (b) call-site review shows no existing in-tree caller relies on the force-strip semantics (all four non-test callers enqueue tasks in 'ready'/non-existent states), but an external consumer could; (c) the fencing design does not require this tightening — D1 correctly fences writes from the stripped lease's generation even if enqueue force-clears, because the force-clear is a dispatcher transaction that takes the global lock and its UPDATE blocks FOR SHARE holders, exactly like release/requeue_stale. Flagged for maintainer decision.

## Consequences

- **Schema**: Postgres migration 3 adds `worker_id TEXT NULL` to `dispatcher_tasks`; sqlite defers (no matching migration this round). No changes to the `events`, `content`, `idempotency`, or `dispatcher_pending_wakes` tables; no changes to partial-index predicates.
- **Postgres EventLog**: `_append` issues an in-tx `SELECT ... FOR SHARE` before consulting `self._lease_validator.is_lease_valid(...)`; the registry is only called when the probe finds no row (mixed wiring / already-fenced lease). The `lease_validator` attribute and `bind_lease_registry` method are retained (other adapters use them; the Protocol still requires the seam). `system_emit(require_lease=False)` is unchanged and does not issue the check.
- **Postgres Dispatcher**: `lease()` writes `worker_id`; `heartbeat()` / `release()` / `release_yield()` / `fail()` / `requeue_stale()` / `fire_due_timers()` / `enqueue()` / `restore_task()` / `purge_task()` clear `worker_id` when clearing `lease_id`. When `now` is not injected, expiry computations use `EXTRACT(EPOCH FROM clock_timestamp())::double precision`; injected `now` retains the test-seam behaviour. Every lifecycle transaction issues `SET LOCAL lock_timeout` (5 s) **after** acquiring the global advisory lock, bounding row-lock waits only: without it, a wedged emit transaction (GC pause, SIGSTOP, dead client) holding the D1 row-share lock could block e.g. a `requeue_stale` UPDATE **while that sweep holds the global dispatcher lock**, stalling every lifecycle op fleet-wide for an unbounded time. On timeout the transaction aborts (`lock_not_available`), the sweep retries next poll, and other tasks' lifecycle ops proceed — a wedged emitter costs one bounded 5 s wait per sweep instead of a fleet stall. `is_lease_valid` is retained for external callers (e.g. observability tooling, test introspection); its result when called from outside an emit transaction is a point-in-time read (not a guarantee), which is acceptable for its callers.
- **Sqlite and InMemory adapters**: no behaviour changes. They continue to use client-side clocks and the `is_lease_valid` seam. Multi-host guarantees do not apply.
- **Worker**: no changes. The worker's InvalidLease handling is already correct:
  - `_HeartbeatRunner._loop` catches `InvalidLease` from `heartbeat()`, logs it, and returns (stops the heartbeat thread) without issuing further writes (worker.py:1349-1367).
  - `WorkerLoop._execute_step` catches `InvalidLease` from `run_leased_task` (which includes any emit's InvalidLease since emits are called deep in the engine/tool path) and logs "relinquishing" without calling `release()` or `fail()` (worker.py:1736-1743). The `except Exception` (non-InvalidLease) branch calls `dispatcher.fail(retryable=True)` in a nested `try` whose own `except Exception` is swallowed (worker.py:1771-1779), so a fail() that itself gets InvalidLease does not recrash.
  - `run_leased_task` never catches InvalidLease itself — it propagates to the WorkerLoop handler. Inside `run_leased_task`, every dispatcher call (release on "skipped"/"drained", _settle_stopped_turn, recovery paths) can raise InvalidLease, and all those propagates go through the same WorkerLoop handler that quietly stops without further writes.
  - Critically, **there is no path where a worker that sees InvalidLease from emit goes on to call fail()**: the InvalidLease from `_append` is raised through `engine.run_one_step` → caught by the `except InvalidLease` in `_execute_step` (line 1736), which does NOT fall through to the `except Exception` fail branch (the `except InvalidLease` is a separate clause above it). So a fenced zombie does exactly what it should: stops writing, does not call fail(), relies on requeue_stale to pick up the pieces.
- **Observability**: `worker_id` on the dispatcher row lets operators see which host holds which lease; the existing ReliabilityEvents continue to be process-local and gain no new fields (they carry `lease_id` already, which is enough to cross-reference).
- **Tests**: New contract tests (see implementation spec) cover cross-adapter-instance zombie emit rejection, requeue-emit races, double-sweeper timer semantics, clock-skew emulation, and worker_id population. Existing contract tests are unchanged.
- **Out of scope**:
  - Multi-host for sqlite or in-memory backends.
  - Member management, service discovery, or cluster membership (any host that can open a Postgres connection to the shared DSN can join; that is a deployment question, not a runtime question).
  - Leader election for any subsystem (all polls are all-hosts-race with advisory-lock serialisation).
  - Tightening `enqueue()` semantics on leased tasks (maintainer decision).
  - Tightening the seed path to prevent other hosts stealing a freshly-enqueued task before the targeted lease (maintainer decision; see spec).
  - Release versioning / changelog beyond the standard patch bump per releasing.md.
